SOLID – Principes de Conception

Pourquoi SOLID ? Réduction dette technique · maintenabilité accrue · testabilité. Corriger : Refactoring.

S – Single Responsibility (SRP) : 1 classe = 1 seule raison de changer. Cible : Classe (cohésion interne).

O – Open/Closed (OCP) : Ouvert extension, fermé modification. Héritage/interfaces pour ajouter comportement sans toucher code source validé.

L – Liskov Substitution (LSP) : Enfant remplace parent sans planter/changer logique. Jamais forcer sous-classe à implémenter méthode inutile.

I – Interface Segregation (ISP) : Plusieurs petites interfaces ciblées > 1 interface géante. Classe ne doit PAS implémenter méthodes inutilisées. Cible : Interface. QCM ★

D – Dependency Inversion (DIP) : Dépendre abstractions, PAS implémentations concrètes. Découpler modules haut niveau des détails techniques.

⚠ ISP ≠ SRP. ISP = segmenter INTERFACES. SRP = 1 classe = 1 responsabilité.

Cadriciels (Frameworks)

Déf : Architecture semi-implémentée, générique. Squelette réutilisable.

Composants : Frozen Spots : Modules concrets immuables. Hot Spots : Modules abstraits à étendre.

IoC / Hollywood : "Ne nous appelez pas, on vous appellera". Le cadre dicte le flux.

Principes : DIP / OCP. Standardisation structure et flux.

Cas : AngularJS (MV*, data-binding) · Django (MVT, sécurité, DRY).

Plugiciels (Plugins)

Déf : Composant ajoute fonction sans modifier système hôte (OCP).

Architecture : Microkernel. Système de base atomique.

Principes : SRP (mission unique) · Inversion Dépendance.

Cas Eclipse : Noyau minimal, tout est plugin. Rétrocompatibilité prioritaire.

Écosystèmes & Breaking Changes

Déf : Collection modules partageant socle commun.

Dilemme coût : Petit changement mainteneur = charge massive utilisateurs.

Stratégies : Mainteneur assume (Eclipse) · Utilisateur assume (Node.js) · Coût partagé (R/CRAN).

AOP – Programmation Orientée Aspect

Déf : Paradigme modélisant fonctionnalités transversales sans affecter classes domaine.

Cross-cutting Concern (CCC) : Logique éparpillée (sécurité, logging).

Problèmes OO : Éparpillement · Confusion · Duplication → Modules moins indépendants.

ÉlémentDéfinition
Join PointPoint précis exécution (méthode, attribut)
PointcutExpression capture Join Points
AdviceCode exécuté (before/after/around)

Weaving : Processus convertit aspects en code compilable. Inputs : Base Code + Aspect Code. Weaver : ajc (AspectJ).

Crosscutting Dynamique : Intercepte événements exécution. before() after() around().

Crosscutting Statique : Modifie structure classes compilation. Ajout membres · Modification hiérarchie · Contrôle compilation.

Applications : Authentification · Logging · Validation · Tests (Mocking) · Conformité architecturale · Design Patterns (17/23 GoF).

Diagramme Styles Architecturaux

MONOLITHIQUE : Mainframe

MODULAIRE : Pipe-Filter · MVC · Blackboard

DISTRIBUÉ : Client-Server · Primary-Secondary · P2P · Broker · SOA · Event-Driven

1. Mainframe (Monolithique)

Déf : Architecture centralisée où traitement, logique et données fusionnés sur 1 unité calcul.

Avantages : Conception simple · contrôle total données · performance élevée · sécurité max.

Désavantages : Évolutivité nulle · maintenance coûteuse · point unique défaillance.

Cas : Systèmes bancaires legacy · Inventaire industriel massif.

2. Pipe-Filter (Modulaire)

Déf : Flux données unidirectionnel où chaque Filtre transforme données et transmet au suivant via Tuyau.

Avantages : Développement parallèle · réutilisabilité · facilité ajout filtres · modules remplaçables.

Désavantages : Surcharge performance · perte efficacité · non interactif.

Cas : Pipeline compilation · Traitement logs Unix (cat|grep|sort).

3. MVC (Modulaire)

Déf : Séparation logique métier (Model), interface (View), orchestration (Controller).

Composants : Model (données + logique) · View (affichage) · Controller (input utilisateur).

Avantages : Développement parallèle · multiples vues · facilité tests · maintenabilité.

Désavantages : Complexité accrue · surcharge petits projets · couplage View-Controller.

Cas : App web e-commerce · App mobile multiplateforme.

4. Blackboard (Modulaire)

Déf : Résolution problèmes complexes sans solution déterministe. Sources connaissances collaborent via espace partagé.

Composants : Blackboard (mémoire partagée) · Sources Connaissances (experts) · Contrôleur (orchestrateur).

Avantages : Flexibilité · tolérance pannes · expérimentation.

Désavantages : Difficile tester/déboguer · pas garantie solution · synchronisation bottleneck.

Cas : Reconnaissance vocale · Diagnostic médical.

5. Client-Server (Distribué)

Déf : Architecture 2 entités : Client (requêtes) et Serveur (services/ressources). QCM ★

Avantages : Centralisation données · contrôle accès · maintenance serveur · scalabilité verticale.

Désavantages : Serveur = point unique défaillance · latence réseau · scalabilité limitée.

Cas : Application bancaire · Serveur email.

6. Primary-Secondary (Distribué)

Déf : Primary décompose tâche et délègue aux Secondary. Coordination centralisée, exécution parallèle.

Avantages : Parallélisme · tolérance pannes · scalabilité horizontale · performance.

Désavantages : Primary = bottleneck · complexité synchronisation · surcharge communication.

Cas : Réplication BD · Calcul distribué (MapReduce).

7. Peer-to-Peer / P2P (Distribué)

Déf : Réseau décentralisé où chaque nœud = client ET serveur. Partage ressources direct sans autorité centrale. QCM ★

Avantages : Scalabilité naturelle · résilience · coûts réduits · autonomie pairs.

Désavantages : Sécurité complexe · découverte pairs difficile · performance variable.

Cas : BitTorrent · Blockchain.

8. Broker (Distribué)

Déf : Intermédiaire coordonne communication entre composants distribués. Découverte services, routage.

Composants : Broker (registre) · Client (demande) · Serveur (fournit, s'enregistre).

Avantages : Découplage fort · ajout/modif dynamique · transparence localisation · load balancing.

Désavantages : Broker = point unique défaillance · latence · complexité gestion.

Cas : CORBA · API Gateway microservices.

9. SOA (Distribué)

Déf : Fonctionnalités exposées comme services indépendants, réutilisables (SOAP/REST/RPC/GraphQL).

Avantages : Réutilisabilité · interopérabilité · scalabilité service par service · agilité métier.

Désavantages : Complexité orchestration · latence réseau · gouvernance difficile · risque cascade pannes.

Cas : Plateforme e-commerce · Microservices modernes.

10. Event-Driven / EDA (Distribué)

Déf : Architecture réactive où composants communiquent via événements asynchrones.

Composants : Producteur (émet) · Bus Événements (Kafka, RabbitMQ) · Consommateur (réagit).

Avantages : Découplage extrême · scalabilité · réactivité temps réel · résilience.

Désavantages : Complexité débogage · ordre événements difficile · duplication possible · latence bus.

Cas : IoT · Notifications utilisateur.

PAGE 3 – ANTI-PATRONS
Définitions Anti-Patrons

Générique : Mauvaise solution à problème courant. Semble efficace au début, cause problèmes majeurs plus tard.

Architectural : Erreur structuration globale système. Mauvais découpage, couplage fort, manque vision.

Conception : Erreur design classes/modules. Abstraction inadéquate, responsabilités confuses.

Code Smell : Symptôme surface indiquant problème profond. Duplication, complexité, nommage.

Anti-Patrons Architecturaux

1. Architecture par Implication : Absence planification. Assume solutions passées s'appliquent. Refactoring : Modularisation vues (GQA).

2. Couvrir ses Arrières : Documentation excessive hyper-détaillée. Refactoring : Abstractions visuelles, communication efficace.

3. Conception par Comité : Trop décideurs sans direction. Refactoring : Rôles clairs, Architecte Principal.

4. Stovepipe System : Migration local→distribué garde hypothèses locales. Refactoring : Patron FAÇADE, abstraction interfaces.

5. Jumble : Mélange chaotique couches horizontales/verticales. Vue avec SQL direct. Refactoring : Abstractions horizontales, implémentations verticales.

6. Reinvent the Wheel : Concevoir de zéro en ignorant standards. Refactoring : Exploration prototypes, processus itératif.

7. Spaghetti Code : Manque structure/cohérence. Code procédural, longues fonctions. Refactoring : Design Patterns, modularisation.

8. Swiss Army Knife : Interface excessivement complexe, trop fonctions. Refactoring : ISP, interfaces ciblées.

9. Vendor Lock-in : Dépendance forte fournisseur spécifique. Refactoring : Abstraction couche, standards ouverts.

Anti-Patrons de Conception

1. Abstraction Manquante : Pas d'interface commune pour entités similaires. Refactoring : Créer classe abstraite/interface.

2. Abstraction Non Utilisée : Interface/classe abstraite jamais utilisée. Refactoring : Supprimer ou implémenter.

3. Multifaceted Abstraction : Classe fait trop choses (viole SRP). Refactoring : Décomposer en classes spécialisées.

4. Hiérarchie Inutilement Profonde : Trop niveaux héritage. Refactoring : Aplatir, composition.

5. Hiérarchie Rebelle : Sous-classe refuse responsabilités parent. Refactoring : Restructurer hiérarchie (LSP).

6. Hiérarchie Brisée : Superclasse dépend sous-classe. Refactoring : Inverser dépendance (DIP).

7. Encapsulation Déficiente : Attributs publics, getters/setters excessifs. Refactoring : Encapsuler, méthodes métier.

MapReduce

Déf : Modèle programmation traitement parallèle données massives. EXAM ★

Phases :

  • Map : Transforme données en paires clé-valeur.
  • Shuffle : Regroupe paires par clé.
  • Reduce : Agrège valeurs par clé.

Avantages : Scalabilité horizontale · tolérance pannes · parallélisme automatique.

Désavantages : Latence élevée (batch) · complexité programmation · surcharge I/O disque.

Cas : Hadoop (traitement batch massif).

Streaming – Architectures

Lambda Architecture : Batch + Speed layers. Batch (précision) + Speed (latence faible). Complexité maintenance 2 systèmes. QCM ★

Kappa Architecture : Stream-only. Simplifie Lambda (1 pipeline). Retraitement = rejouer stream.

Message Brokers

Kafka : Log distribué. Haute débit, persistance, replay. Pub/Sub + streaming. Cas : Event sourcing, pipelines données.

RabbitMQ : Message queue. Routage complexe, garanties livraison. AMQP. Cas : Tâches asynchrones, microservices.

Spark Streaming : Micro-batches. Unifie batch + stream. Latence ~secondes. Cas : Analytics temps quasi-réel.

Théorème CAP

Déf : Système distribué peut garantir MAX 2 sur 3 : EXAM ★

  • Consistency : Tous nœuds voient mêmes données simultanément.
  • Availability : Toute requête reçoit réponse (succès/échec).
  • Partition Tolerance : Système fonctionne malgré coupures réseau.
ChoixGarantiesExemple
CPConsistency + PartitionMongoDB, HBase
APAvailability + PartitionCassandra, DynamoDB
CAConsistency + AvailabilityRDBMS (pas distribué)
⚠ En pratique, P obligatoire (réseau faillible) → choix réel = C ou A.

Déf : Registre distribué immuable. Chaîne blocs cryptographiquement liés. QCM ★

Composants : Bloc (données + hash + hash précédent) · Nœuds (pairs) · Consensus (validation).

Propriétés : Décentralisation · Immuabilité · Transparence · Sécurité cryptographique.

PAGE 5 – SOA · MICROSERVICES · EDA · CONCURRENCE
SOA – Service-Oriented Architecture

Déf : Architecture où fonctionnalités = services indépendants communicants via réseau.

Caractéristiques : Réutilisabilité · Interopérabilité · Découplage · Contrat formel (WSDL).

Composants : Service Provider · Service Consumer · Service Registry (UDDI).

Protocoles : SOAP (XML, lourd, formel) · REST (HTTP, léger, flexible) · RPC · GraphQL (requêtes précises).

Composition : Orchestration (contrôle central, BPEL) · Chorégraphie (décentralisé, événements).

Microservices

Déf : Architecture décompose app en services petits, autonomes, déployables indépendamment. EXAM ★

Caractéristiques : 1 service = 1 fonction métier · BD dédiée · équipe autonome · CI/CD · tolérance pannes.

AspectSOAMicroservices
GranularitéServices largesServices très petits
CommunicationESB (bus central)API directes (REST, gRPC)
DonnéesBD partagée possibleBD par service
GouvernanceCentraliséeDécentralisée

Migration Strangler Fig : Remplacer progressivement monolithe par microservices. Façade route requêtes ancien/nouveau.

Communication : Synchrone (REST, gRPC) · Asynchrone (Message queues, événements).

Patterns : API Gateway · Circuit Breaker · Service Discovery · Saga (transactions distribuées).

Event-Driven Architecture (EDA)

Déf : Architecture où composants réagissent à événements asynchrones.

Patterns Accès Service :

  • Lookup : Client cherche service dans registre. Découplage localisation.
  • Callback : Service notifie client quand prêt. Asynchrone.

Patterns Configuration :

  • Self-Configuration : Service configure lui-même (env vars, fichiers).
  • Centralized Configuration : Config externe (Consul, etcd). Changements dynamiques.

Patterns Synchronisation :

  • Pessimistic : Verrouillage préventif (locks). Cohérence forte, performance réduite.
  • Optimistic : Détection conflits post-facto (versioning). Performance élevée, gestion conflits complexe.
Concurrence – Reactor vs Proactor
AspectReactorProactor
ModèleSynchrone non-bloquantAsynchrone
NotificationÉvénement prêt (readiness)Événement complété (completion)
Qui lit/écritHandler (app)OS (kernel)
ComplexitéMoyenneÉlevée

Reactor : Event loop attend événements → dispatch handlers. Handler fait I/O.

Proactor : App initie I/O asynchrone → OS notifie quand terminé → Handler traite résultat.

💡 Reactor = "Dis-moi quand prêt". Proactor = "Dis-moi quand fini".
Patterns Concurrence Additionnels

Active Object : Découple invocation méthode de son exécution. File requêtes, scheduler, thread dédié.

Half-Sync/Half-Async : Couche sync (haute niveau) + couche async (basse niveau) + queue entre deux. Simplifie programmation.

Leader/Followers : Pool threads. 1 leader attend événements, autres followers attendent. Leader traite → devient follower, nouveau leader élu.

Thread Pool : Réutilise threads pour éviter coût création/destruction. Limite concurrence.

Patterns Résilience

Circuit Breaker : Détecte pannes répétées → ouvre circuit → requêtes échouent vite. États : Closed (normal) · Open (panne) · Half-Open (test récupération).

Retry : Retente opération échouée. Backoff exponentiel recommandé.

Timeout : Limite temps attente. Évite blocages infinis.

Bulkhead : Isole ressources (threads, connexions) par service. Panne 1 service ≠ tout plante.

Fallback : Réponse dégradée si service indisponible (cache, valeur défaut).

Pièges Récurrents Examen
ISP ≠ SRP : ISP segmente INTERFACES pour clients. SRP = 1 classe 1 responsabilité.
P2P : Chaque nœud = client ET serveur. Pas de serveur central.
CAP : En pratique P obligatoire → choix réel C ou A, pas CA distribué.
Microservices ≠ SOA : Microservices = BD par service, déploiement indépendant. SOA = services larges, ESB.
Reactor ≠ Proactor : Reactor = app fait I/O. Proactor = OS fait I/O.
Lambda ≠ Kappa : Lambda = batch + stream. Kappa = stream uniquement.
ACID ≠ BASE : ACID = cohérence forte. BASE = cohérence finale.
PoW ≠ PoS : PoW = puzzle crypto (énergivore). PoS = mise enjeu (éco).
Orchestration ≠ Chorégraphie : Orchestration = contrôle central. Chorégraphie = décentralisé.
Synchrone ≠ Asynchrone : Sync = attend réponse. Async = continue sans attendre.
