SOLID – Principes de conception

Pourquoi SOLID ? Réduction dette technique · maintenabilité accrue · testabilité. Corriger : Refactoring (remonter/descendre méthodes/classes · modifier hiérarchie).

S – Single Responsibility (SRP) : 1 classe = 1 seule raison de changer (1 mission). Truc : Si trop de choses → modifier risque tout casser (effet de bord). Impact : Code + facile à comprendre, modifier, tester (unit tests simples). Cible : Classe (cohésion interne).

O – Open/Closed (OCP) : Ouvert extension, fermé modification. Analogie : Legos (ajouter pièces par-dessus) vs fondre plastique existant. Méthode : Héritage/interfaces pour ajouter comportement sans toucher code source validé.

L – Liskov Substitution (LSP) : Enfant (sous-type) remplace parent (type base) sans planter/changer logique. Test : Véhicule → Voiture, .rouler() se comporte comme attendu. Règle : Jamais forcer sous-classe à implémenter méthode inutile (ex: Autruche hérite Oiseau mais ne peut .voler()).

I – Interface Segregation (ISP) : Plusieurs petites interfaces ciblées > 1 interface géante. Règle : Classe ne doit PAS implémenter méthodes inutilisées. Focus : Découper pour le client (consommateur interface). Cible : Interface (réduire couplage). QCM ★

D – Dependency Inversion (DIP) : Dépendre abstractions (Interfaces/Classes abstraites), PAS implémentations concrètes. Analogie : "Boîte Noire" (on se fiche du comment, tant qu'on sait ce qui sort = contrat). But : Découpler modules haut niveau des détails techniques (changer DB ne casse pas logique affaires).

⚠ Piège exam : ISP ≠ SRP. ISP = segmenter INTERFACES pour clients. SRP = 1 classe = 1 responsabilité (cohésion).

1. Mainframe (Monolithique)

Analogie : Château Fort. Tout à l'intérieur (données, logique, murs). Si pierre tombe → château vulnérable.

Définition : Architecture centralisée où traitement, logique et données fusionnés sur 1 unité calcul.

Avantages : Conception simple · contrôle total données · performance élevée (pas latence comm) · sécurité max contre attaques externes.

Désavantages : Évolutivité nulle (impossible agrandir 1 partie) · maintenance coûteuse (code entrelacé) · point unique défaillance.

Cas : Systèmes bancaires legacy (millions transactions/jour, haute sécurité, aucune dépendance réseau) · Inventaire industriel massif (équipe restreinte, contrôle absolu, frais exécution min).

2. Pipe-Filter (Modulaire)

Analogie : Chaîne Montage. Chaque poste (filtre) transforme produit et passe au suivant par tapis roulant (tuyau).

Définition : Flux données unidirectionnel où chaque Filtre transforme données reçues et transmet au suivant via Tuyau.

Avantages : Développement parallèle (plusieurs dev simultanément) · réutilisabilité composants · facilité ajout filtres · modules remplaçables.

Désavantages : Surcharge performance (traversée chaque filtre) · perte efficacité (formatage données entre modules) · non interactif.

Cas : Pipeline compilation (transformations indépendantes, composants réutilisables) · Traitement logs Unix (cat|grep|sort, flexibilité, tuyaux = synchronisateurs).

3. MVC – Model-View-Controller (Modulaire)

Analogie : Restaurant. Client (Vue) commande → Serveur (Contrôleur) transmet → Cuisine (Modèle) prépare → Serveur ramène plat → Client mange.

Définition : Séparation logique métier (Model), interface utilisateur (View), orchestration (Controller).

Composants : Model (données + logique métier, notifie changements) · View (affichage, écoute Model) · Controller (reçoit input utilisateur, modifie Model/View).

Avantages : Développement parallèle (équipes séparées UI/logique) · multiples vues même modèle · facilité tests unitaires · maintenabilité accrue.

Désavantages : Complexité accrue (3 couches à gérer) · surcharge petits projets · couplage View-Controller parfois fort.

Cas : Application web e-commerce (plusieurs interfaces même données, équipes séparées, évolution UI fréquente) · App mobile multiplateforme (logique partagée, vues iOS/Android distinctes).

4. Blackboard (Modulaire)

Analogie : Tableau Noir Classe. Plusieurs experts écrivent hypothèses → Contrôleur lit → Choisit expert suivant → Répète jusqu'à solution.

Définition : Résolution problèmes complexes sans solution déterministe. Sources connaissances (experts) collaborent via espace partagé (blackboard).

Composants : Blackboard (mémoire partagée, état problème) · Sources Connaissances (modules experts indépendants) · Contrôleur (orchestrateur, sélectionne expert selon état).

Avantages : Flexibilité (ajout/retrait experts facile) · tolérance pannes (1 expert échoue ≠ système plante) · expérimentation (tester stratégies différentes).

Désavantages : Difficile tester/déboguer (comportement émergent) · pas de garantie solution · synchronisation blackboard = bottleneck · complexité contrôleur.

Cas : Reconnaissance vocale (phonétique + syntaxe + sémantique collaborent, pas d'algo unique) · Diagnostic médical (symptômes → experts spécialisés proposent hypothèses, incertitude élevée).

5. Client-Server (Distribué)

Analogie : Restaurant Drive-In. Client commande (requête) → Serveur prépare → Serveur livre (réponse).

Définition : Architecture 2 entités : Client (initie requêtes) et Serveur (fournit services/ressources). Communication réseau.

Avantages : Centralisation données (gestion simplifiée) · contrôle accès sécurisé · maintenance serveur sans toucher clients · scalabilité verticale (améliorer serveur).

Désavantages : Serveur = point unique défaillance · latence réseau · scalabilité limitée (serveur surchargé si trop clients) · coûts infrastructure serveur.

Cas : Application bancaire (sécurité centralisée, transactions cohérentes, clients légers) · Serveur email (stockage centralisé, accès multi-appareils, gestion utilisateurs simplifiée).

6. Primary-Secondary (Distribué)

Analogie : Chef Orchestre. Chef (Primary) distribue partitions → Musiciens (Secondary) jouent leur partie → Harmonie collective.

Définition : Primary décompose tâche et délègue aux Secondary (réplicas/workers). Coordination centralisée, exécution parallèle.

Avantages : Parallélisme (tâches simultanées) · tolérance pannes (si Secondary échoue, Primary réassigne) · scalabilité horizontale (ajout Secondary) · performance accrue.

Désavantages : Primary = bottleneck · complexité synchronisation · surcharge communication Primary-Secondary · pas adapté si tâches interdépendantes.

Cas : Réplication base données (Primary = écriture, Secondary = lecture, haute disponibilité) · Calcul distribué (traitement big data, MapReduce, tâches indépendantes parallélisables).

7. Peer-to-Peer / P2P (Distribué)

Analogie : Potluck Party. Chaque invité apporte plat (ressource) et mange chez autres (consomme). Pas d'hôte central. QCM ★

Définition : Réseau décentralisé où chaque nœud (pair) = client ET serveur. Partage ressources direct sans autorité centrale.

Avantages : Scalabilité naturelle (+ pairs = + ressources) · résilience (pas de point unique défaillance) · coûts réduits (pas serveur central) · autonomie pairs.

Désavantages : Sécurité complexe (pairs non fiables) · découverte pairs difficile · performance variable (dépend pairs disponibles) · synchronisation données ardue.

Cas : Partage fichiers (BitTorrent, pas serveur central, scalabilité infinie) · Blockchain (décentralisation, consensus distribué, résilience attaques).

8. Broker (Distribué)

Analogie : Agent Immobilier. Acheteur cherche maison → Agent (Broker) trouve vendeurs → Met en contact → Transaction.

Définition : Intermédiaire coordonne communication entre composants distribués. Découverte services, routage requêtes, découplage client-serveur.

Composants : Broker (registre services, routage) · Client (demande service) · Serveur (fournit service, s'enregistre au Broker).

Avantages : Découplage fort (client ignore localisation serveur) · ajout/modif services dynamique · transparence localisation · load balancing possible.

Désavantages : Broker = point unique défaillance · latence ajoutée (hop supplémentaire) · complexité gestion Broker · coût infrastructure Broker.

Cas : Middleware entreprise (CORBA, intégration systèmes hétérogènes, découverte services) · API Gateway (microservices, routage intelligent, authentification centralisée).

9. SOA – Service-Oriented Architecture (Distribué)

Analogie : Centre Commercial. Chaque boutique (service) = spécialité. Client visite boutiques nécessaires. Indépendance totale boutiques.

Définition : Architecture où fonctionnalités exposées comme services indépendants, réutilisables, accessibles via réseau (SOAP/REST/RPC/GraphQL).

Avantages : Réutilisabilité services · interopérabilité (langages/plateformes différents) · scalabilité service par service · maintenance isolée · agilité métier.

Désavantages : Complexité orchestration · latence réseau · gouvernance services difficile · coûts infrastructure · dépendances services = risque cascade pannes.

Cas : Plateforme e-commerce (services paiement, inventaire, expédition indépendants, réutilisables) · Microservices modernes (déploiement indépendant, équipes autonomes, CI/CD).

10. Event-Driven / EDA (Distribué)

Analogie : Système Alarme Incendie. Détecteur fumée (producteur) détecte → Alarme sonne (bus événements) → Pompiers réagissent (consommateurs). Découplage total.

Définition : Architecture réactive où composants communiquent via événements asynchrones. Producteurs émettent, consommateurs réagissent via bus d'événements.

Composants : Producteur (émet événements) · Bus Événements (message broker, ex: Kafka, RabbitMQ) · Consommateur (écoute, réagit).

Avantages : Découplage extrême (producteur ignore consommateurs) · scalabilité (ajout consommateurs facile) · réactivité temps réel · résilience (événements persistés).

Désavantages : Complexité débogage (flux asynchrone) · ordre événements difficile garantir · duplication événements possible · latence bus · gestion erreurs complexe.

Cas : IoT (capteurs émettent données → systèmes réagissent temps réel, scalabilité massive) · Notifications utilisateur (action utilisateur → événement → emails/push/SMS, découplage services).

1. CADRICIELS (FRAMEWORKS)

Définition : Architecture semi-implémentée, générique et agnostique. Squelette réutilisable.

Analogie : Centre commercial (Structure fixe + espaces vides).

Composants : Frozen Spots (Modules concrets, fondations, immuables) · Hot Spots (Modules abstraits, emplacements à étendre par le dév).

Inversion de Contrôle (IoC) / Principe Hollywood : "Ne nous appelez pas, on vous appellera". Le cadre dicte le flux. QCM ★

Propriétés & Principes : DIP/OCP (Ouvert extension, fermé modification) · Standardisation (Structure et flux uniformes).

Études de cas : AngularJS (Web JS, Architecture MV*, data-binding automatique, très testable mais complexe) · Django (Python, Architecture MVT Model-View-Template, focus sécurité et développement rapide DRY).

2. PLUGICIELS (PLUGINS)

Définition : Composant qui ajoute une fonction à un système hôte sans le modifier (OCP).

Architecture : Microkernel (Noyau minimal). Le système de base est atomique (fonctionne seul).

Principes clés : SRP (Single Responsibility, principe le plus pertinent, mission unique) · Inversion de Dépendance (système définit interface, plugin l'implémente).

Désavantages : Maintenabilité réduite (fragilité aux mises à jour système) · conflits possibles.

Étude de cas : Eclipse (IDE, noyau minimal où tout est plugin, priorise rétrocompatibilité supporter vieilles versions, complexifie maintenance du cœur).

3. ÉCOSYSTÈMES LOGICIELS & CHANGEMENTS DISRUPTIFS

Définition : Collection de modules partageant un socle commun (Langage, Plateforme).

Propriétés : Interdépendance · Maintenabilité (qualité variable car n'importe qui contribue) · Effet de réseau.

Étude sur la gestion du changement (Breaking Changes) :

Le dilemme du coût : Un petit changement chez le mainteneur peut causer charge de travail massive pour des milliers d'utilisateurs (développeurs tiers).

Stratégies de négociation : 1. Mainteneur assume le coût (Limite changements pour protéger utilisateurs, Eclipse) · 2. Utilisateur assume le coût (Doit s'adapter rapidement aux ruptures, Node.js) · 3. Coût partagé (Collaboration via volontaires et dépôts centralisés, R/CRAN).

Valeurs des communautés spécifiques : Eclipse (Priorité absolue stabilité, accès rapide outils utilisateur final) · R/CRAN (Priorité partage et validation, dépôt distribué paquets statistiques, responsabilité partagée) · Node.js (Priorité vitesse développement, écosystème géré par npm, très dynamique, changements fréquents).

Autre cas : Android (Hybride OS + Cadriciel + Écosystème, Architecture MVVM, Modulaire et sécuritaire).

BIG DATA & NOSQL
I. Définitions et Caractéristiques (Les 3V)

Déf : Un système est considéré "Big Data" s'il répond à au moins l'un des trois V à une échelle où les bases de données traditionnelles (SGBDR) échouent.

  • Volume : Quantité massive de données (Téraoctets, Pétaoctets, Exaoctets).
  • Vélocité : Vitesse de génération, de traitement et d'analyse (temps réel, flux de capteurs, marchés financiers).
  • Variété : Diversité des types de données :
    • Structurées : BD relationnelles classiques.
    • Semi-structurées : Fichiers JSON, XML, logs.
    • Non-structurées : Texte libre, vidéo, images, audio.
II. L'Écosystème NoSQL (Not Only SQL)

Déf : Systèmes de gestion de bases de données distribuées qui n'exposent ni interface SQL standard, ni interface normalisée.

Caractéristiques Fondamentales :

  • Scaling Out (Horizontal) : Au lieu d'augmenter la puissance d'un serveur (Scaling Up), on distribue la charge sur de multiples serveurs bon marché.
  • Schéma Flexible : Sans schéma (schemaless) ou avec schéma flexible pour gérer des données hétérogènes.
  • API Simples : Utilisation d'API propres à chaque technologie (Get, Put, Delete) au lieu d'un langage standardisé.

Théorème CAP (Le Compromis Inévitable) :

Dans un système distribué, on ne peut garantir que 2 des 3 propriétés suivantes simultanément :

  • 1. Consistance (C) : Tous les nœuds voient la même donnée au même moment.
  • 2. Disponibilité (A) : Chaque requête reçoit une réponse (succès ou échec).
  • 3. Tolérance au partitionnement (P) : Le système continue de fonctionner malgré la perte de messages ou d'un nœud.
💡 La plupart des systèmes NoSQL choisissent AP (Disponibilité et Partitionnement) et acceptent la Consistance Éventuelle.
III. Les 4 Modèles de Données NoSQL

1. Clé-Valeur (Key-Value)

  • Déf : Le modèle le plus simple. Une clé unique pointe vers une valeur (opaque pour la BD).
  • Propriétés : Très efficace pour les applications simples ; stockage possible en mémoire vive pour une vitesse maximale.
  • Exemple : Sessions utilisateurs, paniers d'achat, caches.
  • Technologies : Redis, Riak, Amazon DynamoDB.

2. Document (Document Store)

  • Déf : Stocke les données sous forme de documents (fichiers) encodés (JSON, XML, BSON).
  • Propriétés : Les documents peuvent être de tailles et complexités diverses. Les données liées sont souvent imbriquées.
  • Exemple : Profils utilisateurs complexes, catalogues de produits avec attributs variables.
  • Technologies : MongoDB, CouchDB.

3. Colonnes (Wide-Column / Column-Family)

  • Déf : Les données sont stockées par colonnes plutôt que par lignes.
  • Propriétés : Permet de stocker des milliards de lignes avec des millions de colonnes. Très performant pour les agrégations de données massives.
  • Exemple : Historiques de navigation, logs d'événements, séries temporelles.
  • Technologies : Apache Cassandra, HBase.

4. Graphe (Graph Database)

  • Déf : Utilise des structures de graphes avec des nœuds (entités) et des arcs (relations).
  • Propriétés : Optimisé pour traverser les relations complexes. Les propriétés sont stockées sur les nœuds et les arcs.
  • Exemple : Réseaux sociaux (amis de mes amis), systèmes de recommandation, détection de fraude.
  • Technologies : Neo4j, InfiniteGraph.
IV. Analyse et Pipeline de Données

Les Étapes de l'Analyse Big Data :

  • 1. Acquisition : Collecte depuis des sources variées (logs, réseaux sociaux, capteurs).
  • 2. Extraction & Nettoyage : Transformer la donnée brute en format exploitable.
  • 3. Analyse & Modélisation : Découvrir des patrons, faire des prédictions (Data Mining).
  • 4. Interprétation : Visualisation et aide à la décision.

Types d'Analyse :

  • Analyse de Contenu : Traitement de texte (NLP) pour identifier des thèmes ou sentiments.
  • Analyse de Graphe : Mesurer la centralité, l'influence ou détecter des communautés dans un réseau.

ANTI-PATRONS : ARCHITECTURAUX, CONCEPTION & CODE SMELLS
DÉFINITIONS CLÉS

ANTIPATRON (Générique) : "Mauvaise solution" à un problème courant. Semble efficace au début, mais cause problèmes majeurs plus tard. Mots clés : Solution court terme, Dette technique, Piège, Contre-productif.

ANTIPATRON ARCHITECTURAL : Erreur de structuration globale du système (haut niveau). Définit comment les gros composants interagissent. Mots clés : Mauvais découpage, Couplage fort, Manque vision globale, Monolithe rigide, Stovepipe.

ANTIPATRON DE CONCEPTION (Design Smell) : Problèmes de conception au niveau des classes et objets (bas niveau). Rend le code difficile à lire, modifier ou tester. Mots clés : Code smell, Spaghetti, classes trop grosses, Violation SOLID, Manque modularité.

ANTI-PATRONS ARCHITECTURAUX
1. Architecture by Implication

Déf : Absence de planification architecturale explicite. On assume que les solutions passées s'appliquent sans analyse.

Analogie : Bâtir château sur sable car "on sait déjà bâtir châteaux", sans vérifier terrain.

Causes : Manque de temps · Réutilisation aveugle d'anciennes solutions · Absence de documentation.

Symptômes & Conséquences : Aucune doc architecture · Hypothèses implicites non validées · Risque incompatibilité avec nouveaux besoins.

Refactoring : Documenter architecture existante · Valider hypothèses · Créer plan architectural explicite.

Exceptions : Différences entre projets minimes ou très localisées · Projets non critiques ou exploratoires (Prototype/POC).

2. Cover Your Assets (Couvrir ses Arrières)

Déf : Documentation excessive et hyper-détaillée qui tente de lister toutes les exceptions possibles pour éviter les critiques futures.

Analogie : Manuel d'instruction de 1000 pages pour expliquer comment utiliser un marteau.

Causes : Peur des responsabilités ou reproches · Culture d'entreprise bureaucratique · Manque de confiance envers développeurs.

Symptômes & Conséquences : Documentation encombrante, complexe et illisible · Personne ne lit la doc · Perte de temps massive · Architecture réelle dévie de la doc papier.

Refactoring : Abstractions visuelles (diagrammes et tableaux) · Communication efficace (se concentrer sur l'essentiel pour que l'architecture soit comprise, pas juste "écrite").

Exceptions : Projets ultra-critiques (médical, spatial) où chaque détail doit être tracé légalement.

3. Design by Committee (Conception par Comité)

Déf : Trop de décideurs sans direction claire. On essaie de satisfaire tout le monde en fusionnant toutes les idées.

Citations clés : "Un chameau est un cheval dessiné par un comité".

Causes : Rôles mal définis · Volonté de plaire à tout le monde · Réunions trop longues, fréquentes et mal organisées.

Symptômes & Conséquences : Conflits constants entre architectes/développeurs · Doc contradictoire · Architecture incohérente, complexe inutilement · Effort énorme pour interpréter cahier des charges.

Refactoring : Rôles clairs (désigner Architecte Principal = décideur final) · Priorisation (hiérarchiser exigences au lieu de tout inclure) · Gestion de temps (réunions courtes avec objectifs précis, diviser équipes si nécessaire).

Exceptions : Très petites équipes où le consensus naturel est facile et rapide.

4. Stovepipe System (Système en Tuyaux de Poêle)

Déf : Survient lors de la migration d'un système local vers un système distribué. Le code garde des hypothèses "locales" qui ne fonctionnent plus à distance.

Principe brisé : Brise le Single Responsibility et la Dependency Inversion (le métier dépend du système de fichiers).

Causes : Absence de vision globale lors de la distribution · Hypothèses fortes sur l'emplacement des fichiers ou l'accès mémoire local.

Symptômes & Conséquences : Sous-systèmes isolés · Interfaces inappropriées pour le réseau · Impossible de faire communiquer les parties · Bugs de localisation · Système rigide.

Refactoring : Patron FAÇADE (créer interface simplifiée qui cache la complexité et l'isolation des sous-systèmes) · Abstraction des interfaces (redéfinir les interfaces pour qu'elles soient réellement "distribuées").

Exceptions : Systèmes qui n'ont jamais besoin d'être scalables ou distribués (logiciels standalone simples).

5. Le Fouillis (Jumble)

Déf : Mélange chaotique des éléments horizontaux (couches MVC) et verticaux (services/implémentations). QCM ★

Principe brisé : Violation du DIP (Dependency Inversion Principle). On dépend d'implémentations concrètes au lieu d'abstractions.

Causes : Manque de communication entre architectes et développeurs · Développement rapide sans respecter la séparation des responsabilités.

Symptômes & Conséquences : Exemple type : Une Vue qui contient des requêtes SQL directes (court-circuite le Modèle) · Stabilité réduite · Code impossible à réutiliser.

Refactoring : Abstractions horizontales (Définir les couches Architecture comme des interfaces abstraites) · Implémentations verticales (Garder les détails BD, Services comme des implémentations concrètes qui se plient à l'architecture).

6. Réinventer la Roue (Reinvent the Wheel)

Déf : Concevoir un système de zéro en ignorant les architectures ou standards existants.

Principe brisé : Violation de OCP (Open/Closed Principle). L'architecture est trop spécifique et fermée à l'extension.

Causes : "Not Invented Here" syndrome (croire qu'on est les seuls à avoir ce problème) · Connaissance insuffisante des solutions existantes ou des projets passés de la firme.

Symptômes & Conséquences : Application inutile de conceptions existantes (on refait ce qui existe déjà) · Interopérabilité quasi-nulle · Effort de développement démesuré · Maintenance lourde.

Refactoring : Exploration (Identifier les prototypes et architectures standards pertinents avant de coder) · Agilité (Utiliser un processus en spirale/itératif pour raffiner les besoins).

Exceptions : Lorsque la communication est trop difficile ou coûteuse (frais de coordination) : repartir de zéro peut parfois être plus rapide à court terme pour une petite équipe isolée.

7. Spaghetti Code

Déf : Logiciel manquant de structure et de cohérence. Principe brisé : Modularité et abstraction quasi-invisibles.

Causes : Manque d'expérience en conception · Séparation des responsabilités inexistante.

Symptômes & Conséquences : Exemple type : Code procédural, longues fonctions, aucune encapsulation · Compréhension et réutilisation difficiles · Maintenance coûteuse.

Refactoring : Appliquer les patrons de conception (Design Patterns) et les bonnes pratiques · Modulariser le code existant.

Exceptions : Tolérable si inévitable et temporaire, UNIQUEMENT dans l'implémentation (intérieur des fonctions) et non visible dans les interfaces publiques.

8. Swiss Army Knife

Déf : Interface trop complexe et polyvalente, essayant de couvrir trop de besoins différents. QCM ★

Conséquences : Interface chargée, difficile à utiliser et à maintenir.

Refactoring : Diviser l'interface en plusieurs interfaces plus petites et cohérentes (Interface Segregation Principle).

⚠ Swiss Army Knife = interface trop complexe. Jumble = fusion horizontal+vertical. Ne pas confondre.
9. Vendor Lock-in

Déf : Dépendance directe et forte envers un fournisseur spécifique (logiciel ou matériel propriétaire).

Causes : Pas de standard ouvert respecté, format propriétaire · Choix économiques au détriment des techniques.

Symptômes & Conséquences : Mises à niveau fournisseurs pilotent la maintenance, délais induits · Portabilité réduite · Violation de OCP et DIP.

Refactoring : Architecture multiniveau (Séparer les couches pour isoler les détails de l'implémentation) · Abstractions (Créer des interfaces pour ne pas dépendre de technos propriétaires).

ANTI-PATRONS DE CONCEPTION (DESIGN SMELLS)
ANTI-PATRONS DE CONCEPTION : ABSTRACTION

ABSTRACTION : Simplification des entités. Réduction et généralisation des composants pour les rendre plus simples, cohérents et réutilisables.

1. Manque d'Abstraction : Utilisation de types primitifs ou données brutes alors qu'une classe dédiée permettrait de regrouper la logique. Violation SRP. Ex : Utiliser un `string` pour un numéro de téléphone au lieu d'une classe `Telephone`. Refactoring : Extraire une classe.

2. Abstraction à Multifacettes : Une abstraction (classe ou interface) qui possède trop de responsabilités différentes. Violation SRP. Ex : Une interface `DatabaseManager` qui gère la connexion, les requêtes SQL, l'envoi de rapports par mail et l'archivage. Refactoring : Extraire des classes ou super-classes pour séparer chaque responsabilité.

3. Abstraction Dupliquée : Existence de deux abstractions identiques ou presque (clonage). Ex : Deux classes `Client` identiques situées dans deux dossiers différents par erreur. Refactoring : Supprimer le doublon ou extraire le code commun dans une classe partagée/super-classe.

ANTI-PATRONS DE CONCEPTION : ENCAPSULATION

ENCAPSULATION : "Séparation des préoccupations" et "Masquage de l'information".

4. Encapsulation Insuffisante : Une abstraction expose ses détails internes (attributs) plus qu'il n'est nécessaire. Ex : Une classe `CompteBancaire` où l'attribut `solde` est en `public`, permettant de le modifier sans passer par les méthodes `deposer()` ou `retirer()`. Refactoring : Encapsuler les données (passer en `private` ou `protected`) et fournir des méthodes d'accès contrôlées.

5. Encapsulation Inexploitée : Utilisation de vérifications de type explicites (`instanceof`, `if/else` sur les types) au lieu de profiter du polymorphisme. Ex : Une boucle qui vérifie `if (animal instanceof Chien) { aboyer(); } else if (animal instanceof Chat) { miauler(); }`. Refactoring : Remplacer les vérifications de type par le polymorphisme (méthodes virtuelles/abstraites).

ANTI-PATRONS DE CONCEPTION : MODULARITÉ

MODULARITÉ : Capacité d'un système à être divisé en unités distinctes (modules) qui peuvent être créées et modifiées indépendamment. Objectif : Faible couplage (dépendances minimales entre modules) et forte cohésion (chaque module fait une seule chose de façon logique).

6. Modularisation Brisée : Dispersion de données ou méthodes qui devraient être ensemble (similarité sémantique) à travers plusieurs abstractions. Ex : Les données d'un `Utilisateur` sont dans une classe, mais ses méthodes de validation sont dans une autre, et ses préférences dans une troisième sans lien hiérarchique. Refactoring : Regrouper les membres liés dans la même abstraction (encapsulation).

7. Modularisation Insuffisante : Une abstraction est trop vaste ("God Object") ou trop complexe, centralisant trop de membres publics ou de responsabilités. Violation flagrante du SRP. Refactoring : Extraire des méthodes, des classes ou des interfaces plus petites et spécialisées.

8. Modularisation Cycliquement Dépendante : Deux abstractions (ou plus) dépendent l'une de l'autre de manière directe ou indirecte. Refactoring : Casser la boucle en extrayant le code commun dans une troisième classe · Utiliser l'inversion de dépendance (interfaces).

ANTI-PATRONS DE CONCEPTION : HIÉRARCHIE

1. Hiérarchie Non Factorisée : Duplication de code entre les classes dérivées (sœurs) ou entre les classes dérivées et la classe de base. Ex : Classes `Voiture`, `Moto`, `Camion` ont exactement le même code pour la méthode `calculerTaxe()`. Refactoring : Extraire une super-classe · Remonter les méthodes/attributs (Pull up) vers la classe mère.

2. Hiérarchie Cyclique : Une classe de base possède une association directe (dépendance) avec une ou plusieurs de ses classes dérivées. Ex : La classe `Vehicule` (Mère) contient une méthode qui appelle spécifiquement `ClesVoiture` ou `ClesMoto` pour démarrer. Refactoring : Inversion de dépendance (Utiliser une Interface) · Patron Stratégie (Strategy Pattern) · Imbriquer des classes.

3. Hiérarchie Brisée : L'héritage est utilisé uniquement pour récupérer du code existant (réutilisation technique) et non parce qu'il existe un lien logique "EST UN" entre les classes. Ex : `Stack` (Pile) hérite de `Vector`. Problème : Un `Vector` permet d'insérer un élément n'importe où (index 2, 5, etc.). Un `Stack` ne devrait permettre que `push` (ajouter en haut) et `pop` (retirer du haut). Refactoring : Remplacer l'héritage par la délégation · Inverser la relation.

CODE SMELLS : 1. Code Dupliqué

Définition : Même structure de code à plus d'un endroit. Ennemi numéro 1 de la maintenance.

Refactoring : Extract Method (Extraire la méthode) ou Pull Up Field/Method (Remonter vers la classe mère).

// AVANT : Logique de formatage répétée
void imprimerFacture() { System.out.println("--- " + titre.toUpperCase() + " ---"); }
void imprimerRecu() { System.out.println("--- " + titre.toUpperCase() + " ---"); }
// APRÈS : Une seule source de vérité
void imprimerEntete() { System.out.println("--- " + titre.toUpperCase() + " ---"); }

CODE SMELLS : 2-3. Méthode Longue & Longue Liste de Paramètres

2. Méthode Longue : Une méthode qui essaie de trop en faire (souvent > 20-30 lignes). Plus elle est longue, plus elle est difficile à comprendre et à tester. Refactoring : Extract Method. Cherche les commentaires internes; ils indiquent souvent un bloc qui mérite sa propre méthode.

3. Longue Liste de Paramètres : Une méthode qui prend plus de 3 ou 4 paramètres. Difficile à lire et risque d'erreur dans l'ordre des arguments. Refactoring : Introduce Parameter Object.

// AVANT
void creerUtilisateur(String nom, String prenom, String rue, String ville, String cp, String pays) { ... }
// APRÈS
void creerUtilisateur(String nom, String prenom, Adresse adresse) { ... }

CODE SMELLS : 4. Feature Envy

Définition : Une méthode qui utilise plus de membres d'une autre classe que de la classe à laquelle elle appartient.

Refactoring : Move Method (Déplacer la méthode) vers la classe la plus utilisée.

// AVANT (La méthode calculerRemise utilise tout de 'Produit')
class Commande {
  public double calculerRemise(Produit p) {
    return p.getPrixBase() * p.getFacteurTaxe() * 0.10;
  }
}
// APRÈS (La logique est déplacée dans Produit)
class Produit {
  private double prixBase; private double facteurTaxe;
  public double calculerRemise() { return this.prixBase * this.facteurTaxe * 0.10; }
}

CODE SMELLS : 5. Switch Statements

Définition : Présence de `switch` ou de `if/else` massifs basés sur des types de données.

Refactoring : Remplacer les vérifications de type par du Polymorphisme ou le patron Stratégie / État.

// AVANT : Vérifications de type manuelles
void emettreSon(String typeAnimal) {
  if (typeAnimal.equals("Chien")) System.out.println("Woof");
  else if (typeAnimal.equals("Chat")) System.out.println("Meow");
}
// APRÈS : Utilisation du Polymorphisme
abstract class Animal { abstract void crier(); }
class Chien extends Animal { void crier() { System.out.println("Woof"); } }
class Chat extends Animal { void crier() { System.out.println("Meow"); } }

CODE SMELLS : 6-7. Message Chain & Inappropriate Intimacy

6. Message Chain : Un client demande un objet à un autre, qui en demande un autre en cascade (`a.getB().getC().doSomething()`). Refactoring : Hide Delegate (Cacher la délégation).

// AVANT : Couplage fort avec toute la chaîne
client.getDepartement().getGestionnaire().getNom();
// APRÈS : La classe intermédiaire cache la complexité
client.getNomGestionnaire();

7. Inappropriate Intimacy : Une classe accède de manière excessive aux membres privés ou détails internes d'une autre classe. Refactoring : Déplacer des méthodes, extraire des classes ou cacher les délégations.

// AVANT : Voiture manipule directement les internes du Moteur
class Voiture { void accelerer(Moteur m) { m.pressionHuile += 5; m.temperature += 10; } }
// APRÈS : Le moteur gère sa propre logique interne
class Moteur { private int pressionHuile; private double temperature;
  public void monterEnRegime() { this.pressionHuile += 5; this.temperature += 10; } }

CODE SMELLS : 8-9. Divergent Change & Shotgun Surgery

8. Changement Divergent (Divergent Change) : Tu dois modifier une seule classe pour plusieurs raisons différentes (ex: changer la DB et changer l'UI dans le même fichier). Refactoring : Extract Class. Sépare les responsabilités (SRP).

9. Chirurgie au Shotgun (Shotgun Surgery) : L'inverse du divergent : pour faire une seule modification, tu dois modifier 10 classes différentes. Refactoring : Move Method ou Move Field pour regrouper la logique dispersée.

TABLEAU COMPARATIF : Divergent Change vs Shotgun Surgery
OdeurSymptômeRelationSolution
Divergent Change1 Classe → Plusieurs changements différents1 à NExtraire des classes
Shotgun Surgery1 Changement → Plusieurs classes impactéesN à 1Regrouper dans une seule classe
⚠ Note Examen : Si on te dit : "Je veux changer le format de ma base de données et je dois toucher à 12 fichiers différents", la réponse est Shotgun Surgery.
MÉTRIQUES & QUALITÉ

1. Complexité Cyclomatique V(G) : Calcul : V(G) = Nombre de décisions + 1. Interprétation : Un score > 10 indique une méthode trop complexe qui nécessite un refactoring.

2. Métriques de Produit vs Processus : Produit (Mesure le logiciel : Lignes de code, complexité, couplage CBO, cohésion LCOM) · Processus (Mesure le cycle de développement : Nombre de bugs corrigés, temps par tâche).

3. Points Critiques : Cohésion (Force du lien interne. On veut une cohésion FORTE) · Couplage (Interdépendance entre les classes. On veut un couplage FAIBLE).

EDA – Architecture Orientée Événements

Événement – Action pertinente au système (clic, changement données).

Message – Notification suivant l'événement → vers les listeners.

Bus d'événements – Canal/tampon : enregistre, stocke temporairement, transmet aux auditeurs.

4 défis principaux EDA :

1. Accès et configuration de service – configurer l'accès aux interfaces/composants.

2. Synchronisation – accès correct aux ressources partagées.

3. Concurrence – plusieurs tâches sans interférences.

4. Gestion des événements – dispatcher correctement.

Patrons – Accès et Configuration de Service

Wrapper Façade – Encapsule API non-OO dans interfaces OO.

Component Configurator – Bind/unbind des composants à l'exécution sans recompiler.

Intercepteur – Ajoute services transparents à un framework, déclenché par événement spécifique.

Extension Interface – 1 composant exporte plusieurs interfaces → évite l'encombrement.

Patrons de Synchronisation

Scoped Locking – Verrou acquis/libéré auto à l'entrée/sortie d'une portée.

Strategized Locking – Paramétrer le mécanisme de sync d'un composant.

Thread-Safe Interface – Minimise overhead verrou ; évite réacquisition par méthode interne.

Double-Checked Locking – Réduit overhead sync pour sections critiques acquises une seule fois.

Modèles de Concurrence

Active Object – Découple exécution méthode de son invocation (thread propre).

Monitor Object – Une seule méthode s'exécute à la fois au sein d'un objet.

Half-Sync/Half-Async – Découple traitement async et sync pour simplifier sans perte de perf.

Leader/Followers – Threads se relayent sur un pool de sources d'événements : détection → démultiplexage → dispatch → traitement.

Réponses Modèles – Questions Longues

Q3 – Plugin vs Framework :

Plugin : ajoute fonctionnalités sans modifier le core. Flux déclenché par événements, hôte décide quand s'exécute le plugin.

→ Différence clé : qui contrôle le flux ? Plugin = hôte. Framework = framework lui-même (Hollywood Principle : "Don't call us, we'll call you").

Q4.1 – App partage fichiers sans serveur central :

Q4.2 – Patron événement pour app P2P :

→ Proactor : gestion async complète des I/O. Haute concurrence (nombreux transferts simultanés), pas d'attente active, perf optimale pour P2P.

→ RPC : appels directs entre modules, sync, léger (latence min), supporte chiffrement + auth, multi-langages.

AOP – Vocabulaire Complet
TermeDéfinition
Cross-cutting concernPréoccupation qui traverse plusieurs modules (log, cache, sécurité, transactions)
AspectConstruct permettant l'injection d'un cross-cutting concern
AdviceMéthode qui implémente la tâche transversale
PointcutExpression/pattern identifiant OÙ l'Advice est invoqué

WeavingProcessus d'injection des aspects dans le code
Exam réponse : Aspect, Pointcut (pas Joinpoint). Pointcut = pattern (plusieurs points). Joinpoint = instance concrète.
Microservices – Communication Patterns

Async/Non-bloquant (émetteur n'attend pas) :

• Données partagées · Event-driven · Requête-réponse async

Sync/Bloquant (émetteur bloque jusqu'à réponse) :

• Requête-réponse synchrone (REST direct, RPC)

Strangling the Monolith = migration incrémentale : extraire fonctionnalité → nouveau microservice → redirections progressives → monolithe réduit → eventuel remplacement complet.

Derniers Pièges à Ne Pas Rater
1. Blockchain trilemme ≠ CAP
Blockchain : Sécurité + Scalabilité + Décentralisation.
CAP : Consistency + Availability + Partition Tolerance.
2. "Input MapReduce est dynamique" = FAUX
Dataset d'entrée MapReduce = statique (fixé avant le job).
3. Partition Tolerance = obligatoire dans systèmes distribués
On ne peut PAS la sacrifier. On choisit entre C et A.
4. Kafka ≠ "orienté temps"
Kafka = orienté événements. Spark Streaming = orienté temps (micro-batch).
5. ISP ≠ SRP
ISP : découper interfaces pour les clients qui n'utilisent pas tout.
SRP : 1 classe = 1 raison de changer.
6. Proactor ≠ Reactor
Reactor = dispatch sync (attend événement → appelle handler).
Proactor = async complet (lance op → callback sur complétion).
7. SOA : toujours mentionner le protocole !
Si style = SOA → obligatoire de préciser SOAP/REST/RPC/GraphQL.

Interface trop grande → violer ISP ? → Découper en interfaces spécialisées

Streaming orienté événements ? → Kafka (pas Spark)

Streaming seul, simple ? → Kappa

P ne peut pas être sacrifié dans distribué ? → Partition Tolerance (CAP)
