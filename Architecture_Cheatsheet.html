SOLID – Principes de conception

Pourquoi SOLID ? Réduction dette technique · maintenabilité accrue · testabilité. Corriger : Refactoring (remonter/descendre méthodes/classes · modifier hiérarchie).

S – Single Responsibility (SRP) : 1 classe = 1 seule raison de changer (1 mission). Truc : Si trop de choses → modifier risque tout casser (effet de bord). Impact : Code + facile à comprendre, modifier, tester (unit tests simples). Cible : Classe (cohésion interne).

O – Open/Closed (OCP) : Ouvert extension, fermé modification. Analogie : Legos (ajouter pièces par-dessus) vs fondre plastique existant. Méthode : Héritage/interfaces pour ajouter comportement sans toucher code source validé.

L – Liskov Substitution (LSP) : Enfant (sous-type) remplace parent (type base) sans planter/changer logique. Test : Véhicule → Voiture, .rouler() se comporte comme attendu. Règle : Jamais forcer sous-classe à implémenter méthode inutile (ex: Autruche hérite Oiseau mais ne peut .voler()).

I – Interface Segregation (ISP) : Plusieurs petites interfaces ciblées > 1 interface géante. Règle : Classe ne doit PAS implémenter méthodes inutilisées. Focus : Découper pour le client (consommateur interface). Cible : Interface (réduire couplage). QCM ★

D – Dependency Inversion (DIP) : Dépendre abstractions (Interfaces/Classes abstraites), PAS implémentations concrètes. Analogie : "Boîte Noire" (on se fiche du comment, tant qu'on sait ce qui sort = contrat). But : Découpler modules haut niveau des détails techniques (changer DB ne casse pas logique affaires).

⚠ Piège exam : ISP ≠ SRP. ISP = segmenter INTERFACES pour clients. SRP = 1 classe = 1 responsabilité (cohésion).

1. Mainframe (Monolithique)

Analogie : Château Fort. Tout à l'intérieur (données, logique, murs). Si pierre tombe → château vulnérable.

Définition : Architecture centralisée où traitement, logique et données fusionnés sur 1 unité calcul.

Avantages : Conception simple · contrôle total données · performance élevée (pas latence comm) · sécurité max contre attaques externes.

Désavantages : Évolutivité nulle (impossible agrandir 1 partie) · maintenance coûteuse (code entrelacé) · point unique défaillance.

Cas : Systèmes bancaires legacy (millions transactions/jour, haute sécurité, aucune dépendance réseau) · Inventaire industriel massif (équipe restreinte, contrôle absolu, frais exécution min).

2. Pipe-Filter (Modulaire)

Analogie : Chaîne Montage. Chaque poste (filtre) transforme produit et passe au suivant par tapis roulant (tuyau).

Définition : Flux données unidirectionnel où chaque Filtre transforme données reçues et transmet au suivant via Tuyau.

Avantages : Développement parallèle (plusieurs dev simultanément) · réutilisabilité composants · facilité ajout filtres · modules remplaçables.

Désavantages : Surcharge performance (traversée chaque filtre) · perte efficacité (formatage données entre modules) · non interactif.

Cas : Pipeline compilation (transformations indépendantes, composants réutilisables) · Traitement logs Unix (cat|grep|sort, flexibilité, tuyaux = synchronisateurs).

3. MVC – Model-View-Controller (Modulaire)

Analogie : Restaurant. Client (Vue) commande → Serveur (Contrôleur) transmet → Cuisine (Modèle) prépare → Serveur ramène plat → Client mange.

Définition : Séparation logique métier (Model), interface utilisateur (View), orchestration (Controller).

Composants : Model (données + logique métier, notifie changements) · View (affichage, écoute Model) · Controller (reçoit input utilisateur, modifie Model/View).

Avantages : Développement parallèle (équipes séparées UI/logique) · multiples vues même modèle · facilité tests unitaires · maintenabilité accrue.

Désavantages : Complexité accrue (3 couches à gérer) · surcharge petits projets · couplage View-Controller parfois fort.

Cas : Application web e-commerce (plusieurs interfaces même données, équipes séparées, évolution UI fréquente) · App mobile multiplateforme (logique partagée, vues iOS/Android distinctes).

4. Blackboard (Modulaire)

Analogie : Tableau Noir Classe. Plusieurs experts écrivent hypothèses → Contrôleur lit → Choisit expert suivant → Répète jusqu'à solution.

Définition : Résolution problèmes complexes sans solution déterministe. Sources connaissances (experts) collaborent via espace partagé (blackboard).

Composants : Blackboard (mémoire partagée, état problème) · Sources Connaissances (modules experts indépendants) · Contrôleur (orchestrateur, sélectionne expert selon état).

Avantages : Flexibilité (ajout/retrait experts facile) · tolérance pannes (1 expert échoue ≠ système plante) · expérimentation (tester stratégies différentes).

Désavantages : Difficile tester/déboguer (comportement émergent) · pas de garantie solution · synchronisation blackboard = bottleneck · complexité contrôleur.

Cas : Reconnaissance vocale (phonétique + syntaxe + sémantique collaborent, pas d'algo unique) · Diagnostic médical (symptômes → experts spécialisés proposent hypothèses, incertitude élevée).

5. Client-Server (Distribué)

Analogie : Restaurant Drive-In. Client commande (requête) → Serveur prépare → Serveur livre (réponse).

Définition : Architecture 2 entités : Client (initie requêtes) et Serveur (fournit services/ressources). Communication réseau.

Avantages : Centralisation données (gestion simplifiée) · contrôle accès sécurisé · maintenance serveur sans toucher clients · scalabilité verticale (améliorer serveur).

Désavantages : Serveur = point unique défaillance · latence réseau · scalabilité limitée (serveur surchargé si trop clients) · coûts infrastructure serveur.

Cas : Application bancaire (sécurité centralisée, transactions cohérentes, clients légers) · Serveur email (stockage centralisé, accès multi-appareils, gestion utilisateurs simplifiée).

6. Primary-Secondary (Distribué)

Analogie : Chef Orchestre. Chef (Primary) distribue partitions → Musiciens (Secondary) jouent leur partie → Harmonie collective.

Définition : Primary décompose tâche et délègue aux Secondary (réplicas/workers). Coordination centralisée, exécution parallèle.

Avantages : Parallélisme (tâches simultanées) · tolérance pannes (si Secondary échoue, Primary réassigne) · scalabilité horizontale (ajout Secondary) · performance accrue.

Désavantages : Primary = bottleneck · complexité synchronisation · surcharge communication Primary-Secondary · pas adapté si tâches interdépendantes.

Cas : Réplication base données (Primary = écriture, Secondary = lecture, haute disponibilité) · Calcul distribué (traitement big data, MapReduce, tâches indépendantes parallélisables).

7. Peer-to-Peer / P2P (Distribué)

Analogie : Potluck Party. Chaque invité apporte plat (ressource) et mange chez autres (consomme). Pas d'hôte central. QCM ★

Définition : Réseau décentralisé où chaque nœud (pair) = client ET serveur. Partage ressources direct sans autorité centrale.

Avantages : Scalabilité naturelle (+ pairs = + ressources) · résilience (pas de point unique défaillance) · coûts réduits (pas serveur central) · autonomie pairs.

Désavantages : Sécurité complexe (pairs non fiables) · découverte pairs difficile · performance variable (dépend pairs disponibles) · synchronisation données ardue.

Cas : Partage fichiers (BitTorrent, pas serveur central, scalabilité infinie) · Blockchain (décentralisation, consensus distribué, résilience attaques).

8. Broker (Distribué)

Analogie : Agent Immobilier. Acheteur cherche maison → Agent (Broker) trouve vendeurs → Met en contact → Transaction.

Définition : Intermédiaire coordonne communication entre composants distribués. Découverte services, routage requêtes, découplage client-serveur.

Composants : Broker (registre services, routage) · Client (demande service) · Serveur (fournit service, s'enregistre au Broker).

Avantages : Découplage fort (client ignore localisation serveur) · ajout/modif services dynamique · transparence localisation · load balancing possible.

Désavantages : Broker = point unique défaillance · latence ajoutée (hop supplémentaire) · complexité gestion Broker · coût infrastructure Broker.

Cas : Middleware entreprise (CORBA, intégration systèmes hétérogènes, découverte services) · API Gateway (microservices, routage intelligent, authentification centralisée).

9. SOA – Service-Oriented Architecture (Distribué)

Analogie : Centre Commercial. Chaque boutique (service) = spécialité. Client visite boutiques nécessaires. Indépendance totale boutiques.

Définition : Architecture où fonctionnalités exposées comme services indépendants, réutilisables, accessibles via réseau (SOAP/REST/RPC/GraphQL).

Avantages : Réutilisabilité services · interopérabilité (langages/plateformes différents) · scalabilité service par service · maintenance isolée · agilité métier.

Désavantages : Complexité orchestration · latence réseau · gouvernance services difficile · coûts infrastructure · dépendances services = risque cascade pannes.

Cas : Plateforme e-commerce (services paiement, inventaire, expédition indépendants, réutilisables) · Microservices modernes (déploiement indépendant, équipes autonomes, CI/CD).

10. Event-Driven / EDA (Distribué)

Analogie : Système Alarme Incendie. Détecteur fumée (producteur) détecte → Alarme sonne (bus événements) → Pompiers réagissent (consommateurs). Découplage total.

Définition : Architecture réactive où composants communiquent via événements asynchrones. Producteurs émettent, consommateurs réagissent via bus d'événements.

Composants : Producteur (émet événements) · Bus Événements (message broker, ex: Kafka, RabbitMQ) · Consommateur (écoute, réagit).

Avantages : Découplage extrême (producteur ignore consommateurs) · scalabilité (ajout consommateurs facile) · réactivité temps réel · résilience (événements persistés).

Désavantages : Complexité débogage (flux asynchrone) · ordre événements difficile garantir · duplication événements possible · latence bus · gestion erreurs complexe.

Cas : IoT (capteurs émettent données → systèmes réagissent temps réel, scalabilité massive) · Notifications utilisateur (action utilisateur → événement → emails/push/SMS, découplage services).

1. CADRICIELS (FRAMEWORKS)

Définition : Architecture semi-implémentée, générique et agnostique. Squelette réutilisable.

Analogie : Centre commercial (Structure fixe + espaces vides).

Composants : Frozen Spots (Modules concrets, fondations, immuables) · Hot Spots (Modules abstraits, emplacements à étendre par le dév).

Inversion de Contrôle (IoC) / Principe Hollywood : "Ne nous appelez pas, on vous appellera". Le cadre dicte le flux. QCM ★

Propriétés & Principes : DIP/OCP (Ouvert extension, fermé modification) · Standardisation (Structure et flux uniformes).

Études de cas : AngularJS (Web JS, Architecture MV*, data-binding automatique, très testable mais complexe) · Django (Python, Architecture MVT Model-View-Template, focus sécurité et développement rapide DRY).

2. PLUGICIELS (PLUGINS)

Définition : Composant qui ajoute une fonction à un système hôte sans le modifier (OCP).

Architecture : Microkernel (Noyau minimal). Le système de base est atomique (fonctionne seul).

Principes clés : SRP (Single Responsibility, principe le plus pertinent, mission unique) · Inversion de Dépendance (système définit interface, plugin l'implémente).

Désavantages : Maintenabilité réduite (fragilité aux mises à jour système) · conflits possibles.

Étude de cas : Eclipse (IDE, noyau minimal où tout est plugin, priorise rétrocompatibilité supporter vieilles versions, complexifie maintenance du cœur).

3. ÉCOSYSTÈMES LOGICIELS & CHANGEMENTS DISRUPTIFS

Définition : Collection de modules partageant un socle commun (Langage, Plateforme).

Propriétés : Interdépendance · Maintenabilité (qualité variable car n'importe qui contribue) · Effet de réseau.

Étude sur la gestion du changement (Breaking Changes) :

Le dilemme du coût : Un petit changement chez le mainteneur peut causer charge de travail massive pour des milliers d'utilisateurs (développeurs tiers).

Stratégies de négociation : 1. Mainteneur assume le coût (Limite changements pour protéger utilisateurs, Eclipse) · 2. Utilisateur assume le coût (Doit s'adapter rapidement aux ruptures, Node.js) · 3. Coût partagé (Collaboration via volontaires et dépôts centralisés, R/CRAN).

Valeurs des communautés spécifiques : Eclipse (Priorité absolue stabilité, accès rapide outils utilisateur final) · R/CRAN (Priorité partage et validation, dépôt distribué paquets statistiques, responsabilité partagée) · Node.js (Priorité vitesse développement, écosystème géré par npm, très dynamique, changements fréquents).

Autre cas : Android (Hybride OS + Cadriciel + Écosystème, Architecture MVVM, Modulaire et sécuritaire).

BIG DATA & NOSQL
I. Définitions et Caractéristiques (Les 3V)

Déf : Un système est considéré "Big Data" s'il répond à au moins l'un des trois V à une échelle où les bases de données traditionnelles (SGBDR) échouent.

  • Volume : Quantité massive de données (Téraoctets, Pétaoctets, Exaoctets).
  • Vélocité : Vitesse de génération, de traitement et d'analyse (temps réel, flux de capteurs, marchés financiers).
  • Variété : Diversité des types de données :
    • Structurées : BD relationnelles classiques.
    • Semi-structurées : Fichiers JSON, XML, logs.
    • Non-structurées : Texte libre, vidéo, images, audio.
II. L'Écosystème NoSQL (Not Only SQL)

Déf : Systèmes de gestion de bases de données distribuées qui n'exposent ni interface SQL standard, ni interface normalisée.

Caractéristiques Fondamentales :

  • Scaling Out (Horizontal) : Au lieu d'augmenter la puissance d'un serveur (Scaling Up), on distribue la charge sur de multiples serveurs bon marché.
  • Schéma Flexible : Sans schéma (schemaless) ou avec schéma flexible pour gérer des données hétérogènes.
  • API Simples : Utilisation d'API propres à chaque technologie (Get, Put, Delete) au lieu d'un langage standardisé.

Théorème CAP (Le Compromis Inévitable) :

Dans un système distribué, on ne peut garantir que 2 des 3 propriétés suivantes simultanément :

  • 1. Consistance (C) : Tous les nœuds voient la même donnée au même moment.
  • 2. Disponibilité (A) : Chaque requête reçoit une réponse (succès ou échec).
  • 3. Tolérance au partitionnement (P) : Le système continue de fonctionner malgré la perte de messages ou d'un nœud.
💡 La plupart des systèmes NoSQL choisissent AP (Disponibilité et Partitionnement) et acceptent la Consistance Éventuelle.
III. Les 4 Modèles de Données NoSQL

1. Clé-Valeur (Key-Value)

  • Déf : Le modèle le plus simple. Une clé unique pointe vers une valeur (opaque pour la BD).
  • Propriétés : Très efficace pour les applications simples ; stockage possible en mémoire vive pour une vitesse maximale.
  • Exemple : Sessions utilisateurs, paniers d'achat, caches.
  • Technologies : Redis, Riak, Amazon DynamoDB.

2. Document (Document Store)

  • Déf : Stocke les données sous forme de documents (fichiers) encodés (JSON, XML, BSON).
  • Propriétés : Les documents peuvent être de tailles et complexités diverses. Les données liées sont souvent imbriquées.
  • Exemple : Profils utilisateurs complexes, catalogues de produits avec attributs variables.
  • Technologies : MongoDB, CouchDB.

3. Colonnes (Wide-Column / Column-Family)

  • Déf : Les données sont stockées par colonnes plutôt que par lignes.
  • Propriétés : Permet de stocker des milliards de lignes avec des millions de colonnes. Très performant pour les agrégations de données massives.
  • Exemple : Historiques de navigation, logs d'événements, séries temporelles.
  • Technologies : Apache Cassandra, HBase.

4. Graphe (Graph Database)

  • Déf : Utilise des structures de graphes avec des nœuds (entités) et des arcs (relations).
  • Propriétés : Optimisé pour traverser les relations complexes. Les propriétés sont stockées sur les nœuds et les arcs.
  • Exemple : Réseaux sociaux (amis de mes amis), systèmes de recommandation, détection de fraude.
  • Technologies : Neo4j, InfiniteGraph.
IV. Analyse et Pipeline de Données

Les Étapes de l'Analyse Big Data :

  • 1. Acquisition : Collecte depuis des sources variées (logs, réseaux sociaux, capteurs).
  • 2. Extraction & Nettoyage : Transformer la donnée brute en format exploitable.
  • 3. Analyse & Modélisation : Découvrir des patrons, faire des prédictions (Data Mining).
  • 4. Interprétation : Visualisation et aide à la décision.

Types d'Analyse :

  • Analyse de Contenu : Traitement de texte (NLP) pour identifier des thèmes ou sentiments.
  • Analyse de Graphe : Mesurer la centralité, l'influence ou détecter des communautés dans un réseau.

ARCHITECTURES ORIENTÉES SERVICES (SOA)
1. Fondements et Définitions

SOA (Service Oriented Architecture) : Style architectural permettant aux systèmes de collaborer de manière harmonieuse en structurant l'univers en services. Il repose sur la convergence de trois forces : la modularité, l'interopérabilité et l'infrastructure du Web.

LE SERVICE WEB : Composante logicielle atomique et publique qui expose une interface pour fournir une fonctionnalité ou des données.

  • Indépendant : N'exige aucun autre logiciel pour fonctionner.
  • Atomique : Produit des résultats complets.
  • Boîte noire : L'implémentation interne est cachée aux clients.

3. Composantes et Entités SOA

LES ENTITÉS :

  • Fournisseur : Implémente le service, publie l'interface et garantit la qualité (SLA).
  • Client (Consommateur) : Développe l'application-client, génère le stub et suit les termes du fournisseur.
  • Réseau : Infrastructure de communication. Les défaillances réseau incombent au fournisseur et affectent la SLA.

LES ARTEFACTS TECHNIQUES :

  • Interface de service : Document formel (XML/HTML) spécifiant les opérations, paramètres, types de retour et méta-informations (localisation, version).
  • Service Stub : Code généré automatiquement permettant l'accès au réseau et la communication depuis le langage de l'application-client.
  • UDDI : Référentiel global pour la découverte et l'intégration des services.

5. Protocoles et Mécanismes de Communication

RPC (Remote Procedure Call) :

  • Fonctionnement : Étend l'appel local au réseau. Le client sérialise les paramètres (Pack) → Envoi → Le serveur désérialise (Unpack) → Exécution.
  • Avantages : Haute performance, interaction directe.
  • Inconvénients : Couplage étroit, faible découvrabilité.

SOAP (Simple Object Access Protocol) :

  • Structure : Enveloppe XML contenant un Header et un Body.
  • WSDL : Langage XML décrivant les Types, Messages, PortType (opérations), Binding et Service (adresse).
  • Forces : Sécurité entreprise (WS-Security), transactions fiables, gestion d'erreurs normalisée.
  • Faiblesses : Très verbeux (XML), lent (parsing), courbe d'apprentissage abrupte.

REST (Representational State Transfer) :

  • Style : Basé sur les ressources et les méthodes HTTP standards.
  • Méthodes : GET (Lire), POST (Créer), PUT (Mettre à jour), DELETE (Supprimer).
  • Forces : Très scalable, performance élevée (léger), supporte la mise en cache.
  • Faiblesses : Absence de normes formelles (contrats), difficile à composer/rechercher sans standard.

GRAPHQL :

  • Concept : Le client demande exactement les données requises via un seul point d'accès.
  • Caractéristiques : Schéma fortement typé, pas de versionnage (le schéma évolue), support du temps réel (subscriptions).
  • Inconvénient : Mise en cache complexe, risque de sollicitation serveur sur requêtes complexes.

gRPC (Google RPC) :

  • Technologie : Basé sur HTTP/2 et Protobuf (format binaire compact).
  • Forces : Performance maximale (plus rapide que REST), streaming bidirectionnel, génération auto de stubs.
  • Limites : Non supporté nativement par les navigateurs, débogage difficile (binaire).
6. Focus Examen : SLA et Métriques

SLA (Service Level Agreement) : Entente contractuelle sur la qualité du service.

  • Éléments spécifiés : Types de services, méthodes de monitorage, délais de résolution et pénalités.

Métriques de performance critiques :

  • 1. Temps de réponse : Latence pour une demande.
  • 2. Débit (Throughput) : Nombre de demandes par unité de temps.
  • 3. Disponibilité : Taux de temps de fonctionnement (Uptime).
7. Histoire et Évolution (Chronologie)
  • 1991 : CORBA (Partage d'objets via ORB, mais trop complexe/lent).
  • 1998/1999 : XML-RPC / SOAP (Standardisation XML).
  • 2000 : REST (Simplification via HTTP).
  • 2015/2016 : GraphQL / gRPC (Optimisation moderne et performance).
ARCHITECTURES ÉVÉNEMENTIELLES (EDA) & PATRONS DE CONCURRENCE
1. Fondamentaux de l'Architecture Orientée Événements (EDA)

Définitions Clés :

  • Événement : Toute action pertinente au système (ex: clic GUI, changement de données) qui affecte l'état ou le comportement des objets.
  • Message : Notification suivant l'événement, produite par le système/producteur et destinée à l'auditeur.
  • Bus d'événements : Canal/tampon où les événements sont enregistrés, stockés temporairement et transmis.

Propriétés et Avantages :

  • Parallélisation : Capacité augmentée car le bus peut être partagé par plusieurs producteurs/consommateurs.
  • Évolutivité (Scalability) : Les consommateurs peuvent être répliqués pour paralléliser les tâches.
  • Agnotisme Technologique : Interfaces définissables pour différents langages ; base des architectures orientées services (SOA).
  • Réactivité : Traitement asynchrone en temps réel, éliminant les traitements par lots (batch) et réduisant les goulots d'étranglement.

2. Patrons d'Accès et de Configuration de Service

Objectif : Gérer l'accès aux interfaces et l'évolution des composants sans recompilation statique.

A. Wrapper Façade :

  • Définition : Encapsule des fonctions/données de bibliothèques existantes (souvent C non-OO) dans des interfaces OO concises et portables.
  • Exemple : Le Hardware Abstraction Layer (HAL) d'Android encapsulant le Kernel Linux.
  • Avantages : Maintenabilité, modularité, élimination des erreurs de programmation directe sur API bas niveau.
  • Limites : Perte potentielle de fonctionnalités de l'abstraction sous-jacente et légère réduction de performance.

B. Component Configurator (Configurateur de composant) :

  • Définition : Permet de lier/délier des implémentations au moment de l'exécution (runtime) sans redémarrer le processus.
  • Analogie : Un coach de football substituant des joueurs durant un match.
  • Avantages : Configuration dynamique, administration centralisée.
  • Limites : Incertitude accrue sur les interactions entre composants, sécurité réduite.

C. Interceptor (Intercepteur) :

  • Définition : Ajout transparent de services déclenchés automatiquement par des événements spécifiques.
  • Exemple : Réexpédition de courrier lors d'un changement d'adresse.
  • Avantages : Extensibilité, séparation des responsabilités (logging, sécurité).
  • Limites : Point d'insertion de vulnérabilités, risque d'interceptions en cascade.

D. Extension Interface :

  • Définition : Un composant expose plusieurs interfaces pour respecter le principe de ségrégation d'interface et éviter l'encombrement.
  • Mécanisme : Utilisation d'une Root Interface avec getExtension() pour accéder aux fonctionnalités spécifiques.
3. Patrons de Synchronisation

Objectif : Minimiser le surcoût de verrouillage et prévenir l'auto-blocage (deadlock).

A. Scoped Locking (Verrouillage de portée) :

  • Mécanisme : Le verrou est acquis à l'entrée d'un bloc et libéré automatiquement à la sortie (ex: destructeur en C++), même en cas d'exception.
  • Technologie : std::lock_guard en C++.
  • Limite : Risque d'interblocage en cas d'appels récursifs.

B. Strategized Locking (Verrouillage stratégique) :

  • Mécanisme : Paramétrer le mécanisme de synchronisation comme une stratégie pour s'adapter à diverses architectures sans changer le code métier.
  • Avantage : Réutilisabilité et personnalisation selon l'usage (lecture vs écriture).

C. Thread-Safe Interface :

  • Mécanisme :
    • 1. Les méthodes publiques (interface) acquièrent le verrou.
    • 2. Les méthodes privées (implémentation) font confiance aux publiques et n'acquièrent pas de verrou.
  • Avantage : Évite l'auto-blocage lors d'appels intra-composants ou récursifs.

D. Double-Checked Locking Optimization :

  • Objectif : Réduire le surcoût quand une section critique ne doit s'exécuter qu'une seule fois (ex: Singleton).
  • Algorithme : Vérifier un flag/pointeur → Si nul, acquérir le verrou → Re-vérifier (double check) → Exécuter.

5. Gestion des Événements (Event Handling)

Reactor vs Proactor :

  • Reactor (Réactif/Synchrone) : L'application attend l'événement (via un démultiplexeur comme select()) et le traite immédiatement. Principe Hollywood : "Ne nous appelez pas, nous vous appellerons".
  • Proactor (Proactif/Asynchrone) : L'application lance l'opération et continue. Le système (OS) exécute la tâche en arrière-plan et notifie l'application uniquement à la fin via un Completion Handler.

Patrons Complémentaires :

  • Asynchronous Completion Token (ACT) : Également appelé modèle "Cookie". Permet à l'initiateur de passer un jeton au service, qui lui sera retourné inchangé pour identifier le contexte de la réponse.
  • Acceptor-Connector : Découple la connexion/initialisation (Acceptor/Connector) du traitement des données (Service Handler). Essentiel pour les systèmes P2P orientés connexion.
⚡ Focus Examen : Points de vigilance
  • Reactor vs Proactor : Dans le Reactor, c'est l'application qui fait l'I/O après notification. Dans le Proactor, c'est l'OS qui fait l'I/O et l'application reçoit le résultat.
  • Synchronisation : Ne jamais utiliser Scoped Locking seul dans une méthode récursive (risque de deadlock) ; combiner avec Thread-Safe Interface.
  • Performance : Leader/Followers est souvent plus performant que Half-Sync/Half-Async car il évite le passage de données entre threads (pas de file d'attente dynamique).
ANTI-PATRONS : ARCHITECTURAUX, CONCEPTION & CODE SMELLS
DÉFINITIONS CLÉS

ANTIPATRON (Générique) : "Mauvaise solution" à un problème courant. Semble efficace au début, mais cause problèmes majeurs plus tard. Mots clés : Solution court terme, Dette technique, Piège, Contre-productif.

ANTIPATRON ARCHITECTURAL : Erreur de structuration globale du système (haut niveau). Définit comment les gros composants interagissent. Mots clés : Mauvais découpage, Couplage fort, Manque vision globale, Monolithe rigide, Stovepipe.

ANTIPATRON DE CONCEPTION (Design Smell) : Problèmes de conception au niveau des classes et objets (bas niveau). Rend le code difficile à lire, modifier ou tester. Mots clés : Code smell, Spaghetti, classes trop grosses, Violation SOLID, Manque modularité.

ANTI-PATRONS ARCHITECTURAUX
1. Architecture by Implication

Déf : Absence de planification architecturale explicite. On assume que les solutions passées s'appliquent sans analyse.

Analogie : Bâtir château sur sable car "on sait déjà bâtir châteaux", sans vérifier terrain.

Causes : Manque de temps · Réutilisation aveugle d'anciennes solutions · Absence de documentation.

Symptômes & Conséquences : Aucune doc architecture · Hypothèses implicites non validées · Risque incompatibilité avec nouveaux besoins.

Refactoring : Documenter architecture existante · Valider hypothèses · Créer plan architectural explicite.

Exceptions : Différences entre projets minimes ou très localisées · Projets non critiques ou exploratoires (Prototype/POC).

2. Cover Your Assets (Couvrir ses Arrières)

Déf : Documentation excessive et hyper-détaillée qui tente de lister toutes les exceptions possibles pour éviter les critiques futures.

Analogie : Manuel d'instruction de 1000 pages pour expliquer comment utiliser un marteau.

Causes : Peur des responsabilités ou reproches · Culture d'entreprise bureaucratique · Manque de confiance envers développeurs.

Symptômes & Conséquences : Documentation encombrante, complexe et illisible · Personne ne lit la doc · Perte de temps massive · Architecture réelle dévie de la doc papier.

Refactoring : Abstractions visuelles (diagrammes et tableaux) · Communication efficace (se concentrer sur l'essentiel pour que l'architecture soit comprise, pas juste "écrite").

Exceptions : Projets ultra-critiques (médical, spatial) où chaque détail doit être tracé légalement.

3. Design by Committee (Conception par Comité)

Déf : Trop de décideurs sans direction claire. On essaie de satisfaire tout le monde en fusionnant toutes les idées.

Citations clés : "Un chameau est un cheval dessiné par un comité".

Causes : Rôles mal définis · Volonté de plaire à tout le monde · Réunions trop longues, fréquentes et mal organisées.

Symptômes & Conséquences : Conflits constants entre architectes/développeurs · Doc contradictoire · Architecture incohérente, complexe inutilement · Effort énorme pour interpréter cahier des charges.

Refactoring : Rôles clairs (désigner Architecte Principal = décideur final) · Priorisation (hiérarchiser exigences au lieu de tout inclure) · Gestion de temps (réunions courtes avec objectifs précis, diviser équipes si nécessaire).

Exceptions : Très petites équipes où le consensus naturel est facile et rapide.

4. Stovepipe System (Système en Tuyaux de Poêle)

Déf : Survient lors de la migration d'un système local vers un système distribué. Le code garde des hypothèses "locales" qui ne fonctionnent plus à distance.

Principe brisé : Brise le Single Responsibility et la Dependency Inversion (le métier dépend du système de fichiers).

Causes : Absence de vision globale lors de la distribution · Hypothèses fortes sur l'emplacement des fichiers ou l'accès mémoire local.

Symptômes & Conséquences : Sous-systèmes isolés · Interfaces inappropriées pour le réseau · Impossible de faire communiquer les parties · Bugs de localisation · Système rigide.

Refactoring : Patron FAÇADE (créer interface simplifiée qui cache la complexité et l'isolation des sous-systèmes) · Abstraction des interfaces (redéfinir les interfaces pour qu'elles soient réellement "distribuées").

Exceptions : Systèmes qui n'ont jamais besoin d'être scalables ou distribués (logiciels standalone simples).

5. Le Fouillis (Jumble)

Déf : Mélange chaotique des éléments horizontaux (couches MVC) et verticaux (services/implémentations). QCM ★

Principe brisé : Violation du DIP (Dependency Inversion Principle). On dépend d'implémentations concrètes au lieu d'abstractions.

Causes : Manque de communication entre architectes et développeurs · Développement rapide sans respecter la séparation des responsabilités.

Symptômes & Conséquences : Exemple type : Une Vue qui contient des requêtes SQL directes (court-circuite le Modèle) · Stabilité réduite · Code impossible à réutiliser.

Refactoring : Abstractions horizontales (Définir les couches Architecture comme des interfaces abstraites) · Implémentations verticales (Garder les détails BD, Services comme des implémentations concrètes qui se plient à l'architecture).

6. Réinventer la Roue (Reinvent the Wheel)

Déf : Concevoir un système de zéro en ignorant les architectures ou standards existants.

Principe brisé : Violation de OCP (Open/Closed Principle). L'architecture est trop spécifique et fermée à l'extension.

Causes : "Not Invented Here" syndrome (croire qu'on est les seuls à avoir ce problème) · Connaissance insuffisante des solutions existantes ou des projets passés de la firme.

Symptômes & Conséquences : Application inutile de conceptions existantes (on refait ce qui existe déjà) · Interopérabilité quasi-nulle · Effort de développement démesuré · Maintenance lourde.

Refactoring : Exploration (Identifier les prototypes et architectures standards pertinents avant de coder) · Agilité (Utiliser un processus en spirale/itératif pour raffiner les besoins).

Exceptions : Lorsque la communication est trop difficile ou coûteuse (frais de coordination) : repartir de zéro peut parfois être plus rapide à court terme pour une petite équipe isolée.

7. Spaghetti Code

Déf : Logiciel manquant de structure et de cohérence. Principe brisé : Modularité et abstraction quasi-invisibles.

Causes : Manque d'expérience en conception · Séparation des responsabilités inexistante.

Symptômes & Conséquences : Exemple type : Code procédural, longues fonctions, aucune encapsulation · Compréhension et réutilisation difficiles · Maintenance coûteuse.

Refactoring : Appliquer les patrons de conception (Design Patterns) et les bonnes pratiques · Modulariser le code existant.

Exceptions : Tolérable si inévitable et temporaire, UNIQUEMENT dans l'implémentation (intérieur des fonctions) et non visible dans les interfaces publiques.

8. Swiss Army Knife

Déf : Interface trop complexe et polyvalente, essayant de couvrir trop de besoins différents. QCM ★

Conséquences : Interface chargée, difficile à utiliser et à maintenir.

Refactoring : Diviser l'interface en plusieurs interfaces plus petites et cohérentes (Interface Segregation Principle).

⚠ Swiss Army Knife = interface trop complexe. Jumble = fusion horizontal+vertical. Ne pas confondre.
9. Vendor Lock-in

Déf : Dépendance directe et forte envers un fournisseur spécifique (logiciel ou matériel propriétaire).

Causes : Pas de standard ouvert respecté, format propriétaire · Choix économiques au détriment des techniques.

Symptômes & Conséquences : Mises à niveau fournisseurs pilotent la maintenance, délais induits · Portabilité réduite · Violation de OCP et DIP.

Refactoring : Architecture multiniveau (Séparer les couches pour isoler les détails de l'implémentation) · Abstractions (Créer des interfaces pour ne pas dépendre de technos propriétaires).

ANTI-PATRONS DE CONCEPTION (DESIGN SMELLS)
ANTI-PATRONS DE CONCEPTION : ABSTRACTION

ABSTRACTION : Simplification des entités. Réduction et généralisation des composants pour les rendre plus simples, cohérents et réutilisables.

1. Manque d'Abstraction : Utilisation de types primitifs ou données brutes alors qu'une classe dédiée permettrait de regrouper la logique. Violation SRP. Ex : Utiliser un `string` pour un numéro de téléphone au lieu d'une classe `Telephone`. Refactoring : Extraire une classe.

2. Abstraction à Multifacettes : Une abstraction (classe ou interface) qui possède trop de responsabilités différentes. Violation SRP. Ex : Une interface `DatabaseManager` qui gère la connexion, les requêtes SQL, l'envoi de rapports par mail et l'archivage. Refactoring : Extraire des classes ou super-classes pour séparer chaque responsabilité.

3. Abstraction Dupliquée : Existence de deux abstractions identiques ou presque (clonage). Ex : Deux classes `Client` identiques situées dans deux dossiers différents par erreur. Refactoring : Supprimer le doublon ou extraire le code commun dans une classe partagée/super-classe.

ANTI-PATRONS DE CONCEPTION : ENCAPSULATION

ENCAPSULATION : "Séparation des préoccupations" et "Masquage de l'information".

4. Encapsulation Insuffisante : Une abstraction expose ses détails internes (attributs) plus qu'il n'est nécessaire. Ex : Une classe `CompteBancaire` où l'attribut `solde` est en `public`, permettant de le modifier sans passer par les méthodes `deposer()` ou `retirer()`. Refactoring : Encapsuler les données (passer en `private` ou `protected`) et fournir des méthodes d'accès contrôlées.

5. Encapsulation Inexploitée : Utilisation de vérifications de type explicites (`instanceof`, `if/else` sur les types) au lieu de profiter du polymorphisme. Ex : Une boucle qui vérifie `if (animal instanceof Chien) { aboyer(); } else if (animal instanceof Chat) { miauler(); }`. Refactoring : Remplacer les vérifications de type par le polymorphisme (méthodes virtuelles/abstraites).

ANTI-PATRONS DE CONCEPTION : MODULARITÉ

MODULARITÉ : Capacité d'un système à être divisé en unités distinctes (modules) qui peuvent être créées et modifiées indépendamment. Objectif : Faible couplage (dépendances minimales entre modules) et forte cohésion (chaque module fait une seule chose de façon logique).

6. Modularisation Brisée : Dispersion de données ou méthodes qui devraient être ensemble (similarité sémantique) à travers plusieurs abstractions. Ex : Les données d'un `Utilisateur` sont dans une classe, mais ses méthodes de validation sont dans une autre, et ses préférences dans une troisième sans lien hiérarchique. Refactoring : Regrouper les membres liés dans la même abstraction (encapsulation).

7. Modularisation Insuffisante : Une abstraction est trop vaste ("God Object") ou trop complexe, centralisant trop de membres publics ou de responsabilités. Violation flagrante du SRP. Refactoring : Extraire des méthodes, des classes ou des interfaces plus petites et spécialisées.

8. Modularisation Cycliquement Dépendante : Deux abstractions (ou plus) dépendent l'une de l'autre de manière directe ou indirecte. Refactoring : Casser la boucle en extrayant le code commun dans une troisième classe · Utiliser l'inversion de dépendance (interfaces).

ANTI-PATRONS DE CONCEPTION : HIÉRARCHIE

1. Hiérarchie Non Factorisée : Duplication de code entre les classes dérivées (sœurs) ou entre les classes dérivées et la classe de base. Ex : Classes `Voiture`, `Moto`, `Camion` ont exactement le même code pour la méthode `calculerTaxe()`. Refactoring : Extraire une super-classe · Remonter les méthodes/attributs (Pull up) vers la classe mère.

2. Hiérarchie Cyclique : Une classe de base possède une association directe (dépendance) avec une ou plusieurs de ses classes dérivées. Ex : La classe `Vehicule` (Mère) contient une méthode qui appelle spécifiquement `ClesVoiture` ou `ClesMoto` pour démarrer. Refactoring : Inversion de dépendance (Utiliser une Interface) · Patron Stratégie (Strategy Pattern) · Imbriquer des classes.

3. Hiérarchie Brisée : L'héritage est utilisé uniquement pour récupérer du code existant (réutilisation technique) et non parce qu'il existe un lien logique "EST UN" entre les classes. Ex : `Stack` (Pile) hérite de `Vector`. Problème : Un `Vector` permet d'insérer un élément n'importe où (index 2, 5, etc.). Un `Stack` ne devrait permettre que `push` (ajouter en haut) et `pop` (retirer du haut). Refactoring : Remplacer l'héritage par la délégation · Inverser la relation.

CODE SMELLS : 1. Code Dupliqué

Définition : Même structure de code à plus d'un endroit. Ennemi numéro 1 de la maintenance.

Refactoring : Extract Method (Extraire la méthode) ou Pull Up Field/Method (Remonter vers la classe mère).

// AVANT : Logique de formatage répétée
void imprimerFacture() { System.out.println("--- " + titre.toUpperCase() + " ---"); }
void imprimerRecu() { System.out.println("--- " + titre.toUpperCase() + " ---"); }
// APRÈS : Une seule source de vérité
void imprimerEntete() { System.out.println("--- " + titre.toUpperCase() + " ---"); }

CODE SMELLS : 2-3. Méthode Longue & Longue Liste de Paramètres

2. Méthode Longue : Une méthode qui essaie de trop en faire (souvent > 20-30 lignes). Plus elle est longue, plus elle est difficile à comprendre et à tester. Refactoring : Extract Method. Cherche les commentaires internes; ils indiquent souvent un bloc qui mérite sa propre méthode.

3. Longue Liste de Paramètres : Une méthode qui prend plus de 3 ou 4 paramètres. Difficile à lire et risque d'erreur dans l'ordre des arguments. Refactoring : Introduce Parameter Object.

// AVANT
void creerUtilisateur(String nom, String prenom, String rue, String ville, String cp, String pays) { ... }
// APRÈS
void creerUtilisateur(String nom, String prenom, Adresse adresse) { ... }

CODE SMELLS : 4. Feature Envy

Définition : Une méthode qui utilise plus de membres d'une autre classe que de la classe à laquelle elle appartient.

Refactoring : Move Method (Déplacer la méthode) vers la classe la plus utilisée.

// AVANT (La méthode calculerRemise utilise tout de 'Produit')
class Commande {
  public double calculerRemise(Produit p) {
    return p.getPrixBase() * p.getFacteurTaxe() * 0.10;
  }
}
// APRÈS (La logique est déplacée dans Produit)
class Produit {
  private double prixBase; private double facteurTaxe;
  public double calculerRemise() { return this.prixBase * this.facteurTaxe * 0.10; }
}

CODE SMELLS : 5. Switch Statements

Définition : Présence de `switch` ou de `if/else` massifs basés sur des types de données.

Refactoring : Remplacer les vérifications de type par du Polymorphisme ou le patron Stratégie / État.

// AVANT : Vérifications de type manuelles
void emettreSon(String typeAnimal) {
  if (typeAnimal.equals("Chien")) System.out.println("Woof");
  else if (typeAnimal.equals("Chat")) System.out.println("Meow");
}
// APRÈS : Utilisation du Polymorphisme
abstract class Animal { abstract void crier(); }
class Chien extends Animal { void crier() { System.out.println("Woof"); } }
class Chat extends Animal { void crier() { System.out.println("Meow"); } }

CODE SMELLS : 6-7. Message Chain & Inappropriate Intimacy

6. Message Chain : Un client demande un objet à un autre, qui en demande un autre en cascade (`a.getB().getC().doSomething()`). Refactoring : Hide Delegate (Cacher la délégation).

// AVANT : Couplage fort avec toute la chaîne
client.getDepartement().getGestionnaire().getNom();
// APRÈS : La classe intermédiaire cache la complexité
client.getNomGestionnaire();

7. Inappropriate Intimacy : Une classe accède de manière excessive aux membres privés ou détails internes d'une autre classe. Refactoring : Déplacer des méthodes, extraire des classes ou cacher les délégations.

// AVANT : Voiture manipule directement les internes du Moteur
class Voiture { void accelerer(Moteur m) { m.pressionHuile += 5; m.temperature += 10; } }
// APRÈS : Le moteur gère sa propre logique interne
class Moteur { private int pressionHuile; private double temperature;
  public void monterEnRegime() { this.pressionHuile += 5; this.temperature += 10; } }

CODE SMELLS : 8-9. Divergent Change & Shotgun Surgery

8. Changement Divergent (Divergent Change) : Tu dois modifier une seule classe pour plusieurs raisons différentes (ex: changer la DB et changer l'UI dans le même fichier). Refactoring : Extract Class. Sépare les responsabilités (SRP).

9. Chirurgie au Shotgun (Shotgun Surgery) : L'inverse du divergent : pour faire une seule modification, tu dois modifier 10 classes différentes. Refactoring : Move Method ou Move Field pour regrouper la logique dispersée.

MÉTRIQUES & QUALITÉ

1. Complexité Cyclomatique V(G) : Calcul : V(G) = Nombre de décisions + 1. Interprétation : Un score > 10 indique une méthode trop complexe qui nécessite un refactoring.

2. Métriques de Produit vs Processus : Produit (Mesure le logiciel : Lignes de code, complexité, couplage CBO, cohésion LCOM) · Processus (Mesure le cycle de développement : Nombre de bugs corrigés, temps par tâche).

3. Points Critiques : Cohésion (Force du lien interne. On veut une cohésion FORTE) · Couplage (Interdépendance entre les classes. On veut un couplage FAIBLE).

BLOCKCHAIN
1. Concepts Fondamentaux et Définition

La Blockchain est un registre numérique distribué, immuable et basé sur le consensus, utilisant un réseau Peer-to-Peer (P2P) pour sécuriser des données sans autorité centrale.

Propriétés Clés :

  • Immuabilité : Les données sont permanentes et inaltérables ; on ne peut qu'actualiser l'état du système via de nouvelles transactions.
  • Transparence : Registre public où toutes les données sont visibles par les participants.
  • Décentralisation : Copies du registre distribuées sur le réseau ; pas de point de défaillance unique.
  • Consensus : Mécanisme de vérification de la confiance pour valider l'état du système.
2. Architecture et Fonctionnement

Structure des Blocs et Transactions :

  • Bloc : Contient plusieurs transactions validées cryptographiquement.
  • Liaison : Chaque bloc contient le Hash du bloc précédent (chaînage), rendant toute modification rétroactive virtuellement impossible sans recalculer toute la chaîne.

Processus d'ajout :

  • 1. Demande/Authentification d'une transaction
  • 2. Création d'un bloc de transaction
  • 3. Diffusion à chaque nœud du réseau
  • 4. Validation par les nœuds (Consensus)
  • 5. Ajout à la blockchain existante et distribution de la mise à jour

3. Le Problème des Généraux Byzantins (BFT)

Problème de théorie des jeux : Comment des parties dispersées peuvent-elles atteindre un consensus sans autorité centrale, sachant que certains acteurs peuvent être des traîtres?

Paramètres Critiques pour l'Examen :

  • Condition de Victoire : Nécessite que 2/3 des généraux attaquent simultanément.
  • Seuil de Tolérance (3f+1) :
    • f = nombre de traîtres
    • Nombre minimum de généraux nécessaires pour garantir le consensus : 3f + 1
    • Exemple : Pour tolérer 1 traître, il faut au moins 4 généraux (3(1)+1 = 4)
4. Mécanismes de Consensus : PoW vs PoS

Proof-of-Work (PoW) - "Minage"

  • Principe : Résoudre une énigme computationnelle complexe (SHA-256).
  • Fonctionnement : Trouver un nonce tel que le Hash du bloc soit ≤ à une valeur cible (Target).
  • Avantage : Sécurité robuste, première implémentation réelle de la tolérance aux pannes byzantines à grande échelle.
  • Inconvénients : Énorme consommation électrique, faible scalabilité (ex: 7 TPS pour Bitcoin).

Proof-of-Stake (PoS) - "Forgeage"

  • Principe : Sélection des validateurs selon le montant de jetons "stakés" (verrouillés en garantie).
  • Avantages : Efficacité énergétique, meilleure scalabilité.
  • Vulnérabilité (Nothing-at-stake) : Les validateurs peuvent voter sur plusieurs forks simultanément sans coût, affaiblissant la sécurité.

7. Smart Contracts (Contrats Intelligents)

Morceau de code et de données déployé sur une blockchain qui applique automatiquement les termes d'un accord.

Caractéristiques Techniques :

  • Déterministe : Même entrée = même résultat pour tous les participants (cohérence du réseau).
  • Immuable : Une fois déployé, ne peut être ni modifié ni supprimé.
  • Vérifiable : Possède une adresse unique sur la blockchain.
  • Gas (Ethereum) : Chaque instruction a un coût prédéterminé pour limiter l'utilisation des ressources.

Cycle d'Exécution :

  • 1. Contrat prédéfini : Termes codés et immuables
  • 2. Événement : Déclencheur (ex: date, réception de données)
  • 3. Exécution : Transfert automatique de valeur
  • 4. Règlement : Paiement instantané et efficace
8. Sécurité et Attaques

Attaque 51% (PoW) :

  • Définition : Un groupe de mineurs contrôle >50% de la puissance de hachage du réseau.
  • Capacités de l'attaquant : Double dépense (effacer une transaction), bloquer des transactions, réordonner les blocs récents.
  • Limites : Ne peut pas créer de nouvelles pièces, voler des pièces existantes ou altérer les très vieux blocs.

Signature Numérique :

  • Expéditeur : Hache la transaction + chiffre avec sa clé privée.
  • Vérification : Les nœuds utilisent la clé publique de l'expéditeur. Si le hash correspond, l'intégrité et l'identité sont prouvées.

1. Définitions et Fondamentaux

Qu'est-ce qu'un Microservice ?

  • Définition Académique : Une version plus petite de la SOA (Service-Oriented Architecture) où chaque service est un processus unique avec une seule responsabilité (Principe Unix).

2. Le Monolithe : Avantages et Désavantages

Propriétés :

  • Processus unique : Un monolithe n'est pas un seul programme, mais un processus unique en exécution (peut être modulaire, mais pas distribué).
  • Architecture 3 couches : Client, Serveur, Base de données.

Analyse Critique :

  • Avantages : Équipe unique productive, facile à tester/déployer (fichiers WAR), supporté nativement par les IDE, mise à l'échelle horizontale simple via équilibreur de charge.
  • Inconvénients (Loi de la croissance) :
    • Surcharge des conteneurs (Tomcat) et IDE
    • Frontières floues : Modularité facile à briser involontairement
    • Verrouillage (Vendor Lock-in) : Engagement long terme envers une pile technologique (ex: tout Java/Tomcat)
    • Déploiement rigide : Un seul changement force le redéploiement de tout le système
3. Caractéristiques des Microservices (9 Principes)
  • 1. Composantes via services : Unités remplaçables et évolutives indépendamment.
  • 2. Produits et non projets : Mentalité "Vous le construisez, vous l'exécutez" (Amazon). L'équipe est responsable du logiciel en production.
  • 3. Points de terminaison intelligents / Tuyaux stupides : La logique réside chez l'expéditeur/récepteur (HTTP/REST), pas dans le réseau (contrairement aux ESB de la SOA).
  • 4. Gouvernance décentralisée : Liberté du choix de la pile technologique par service.
  • 5. Gestion décentralisée des données : Un domaine complexe est divisé en domaines délimités (Bounded Contexts - DDD). Chaque service possède sa propre BD.
  • 6. Automatisation de l'infrastructure : Investissement massif dans les pipelines CI/CD (Testing et Déploiement "ennuyeux").
  • 7. Concevoir pour l'échec : Monitoring technique et commercial impératif. Si un service échoue, il doit le faire gracieusement.
  • 8. Conception évolutive : Possibilité de remplacer un composant sans affecter le système.
  • 9. Loi de Conway : La structure du logiciel reflète la structure de communication de l'organisation.
4. Stratégies de Migration et Conception

Patron Strangler Application (Étouffement) :

  • Mécanisme : Le monolithe reste le cœur initial. Les nouvelles fonctionnalités sont ajoutées comme microservices. On remplace progressivement les fonctions de base du monolithe par des microservices jusqu'à disparition complète du monolithe.
  • Règle Shotgun Surgery : Si deux services sont modifiés ensemble de façon répétée, ils doivent probablement être fusionnés.

Processus d'Assemblage (Chris Richardson) :

  • Entrée : User stories, exigences non fonctionnelles (SLA), Wireframes.
  • Processus : Définir les opérations système → Définir les sous-domaines → Grouper les sous-domaines en services → Concevoir les collaborations.
  • Résultat : Schémas d'architecture, contrats d'API, modèles de Bounded Contexts.

6. Focus Examen : Antipatrons d'Adoption
  • Magic Pixie Dust : Croire que les microservices résoudront tous les problèmes sans changer la culture (DevOps).
  • Microservices as the Goal : Adopter l'architecture pour la technologie plutôt que pour régler un problème d'affaires.
  • Scattershot Adoption : Adoption sans stratégie centrale, chaque équipe fait ses propres choix (manque de coordination).
  • Trying to fly before walking : Adopter les microservices sans maîtriser le code propre, les tests automatisés et le CI/CD.
  • More the Merrier : Créer des services trop petits (Nanoservices), ce qui explose la complexité de communication.
  • Red Flag Law : Utiliser des microservices avec des processus lents (Cascade, tests manuels).
7. Étude de cas : Amazon Prime Video (Retour au Monolithe)
  • Contexte : Passage d'une architecture microservices distribuée (Lambda + Step Functions) à un monolithe en 2023.
  • Problème : Goulot d'étranglement dû aux transferts de données excessifs vers S3, latence élevée et coûts d'I/O.
  • Solution : Regroupement de tous les composants (détecteurs, convertisseurs) dans un seul processus (Amazon ECS).
  • Leçon : Il n'y a pas de modèle universel. L'architecture doit être réévaluée à chaque ordre de grandeur de croissance.
AOP – PROGRAMMATION ORIENTÉE ASPECTS
I. Fondamentaux et Paradigme

Définitions Clés :

  • AOP (Aspect-Oriented Programming) : Paradigme de programmation visant à modéliser les fonctionnalités transversales sans affecter les classes du domaine.
  • Fonctionnalité Transversale (Cross-cutting Concern - CCC) : Logique qui s'éparpille dans plusieurs modules ou classes (ex: sécurité, logging).
  • Niveau d'abstraction : L'AOP est plus abstrait que le paradigme Orienté Objet (OO), se situant au-dessus dans la hiérarchie des langages.

Problème résolu par l'AOP :

  • Code Tangling (Enchevêtrement) : Plusieurs préoccupations mélangées dans une même classe.
  • Code Scattering (Éparpillement) : Une préoccupation dispersée dans plusieurs modules.
II. Éléments du Modèle AOP
  • Join Point : Point d'exécution dans le programme (ex: appel de méthode, accès à un attribut).
  • Pointcut : Expression qui sélectionne un ensemble de Join Points où un Advice sera appliqué. QCM ★
  • Advice : Code à exécuter à un Join Point (Before, After, Around).
  • Aspect : Module qui encapsule un Pointcut et un Advice pour modéliser une préoccupation transversale. QCM ★
  • Weaving (Tissage) : Processus d'intégration des Aspects dans le code de base (compile-time, load-time, runtime).
⚠ Réponse QCM : Aspect, Pointcut (option b). Joinpoint = point d'exécution spécifique, Pointcut = pattern/expression qui matche plusieurs joinpoints.
III. Types de Crosscutting
  • Homogène : Même code injecté partout (ex: logging identique dans toutes les méthodes).
  • Hétérogène : Code différent selon le contexte (ex: validation différente par méthode).
IV. Domaines d'Application
  • Logging : Tracer les appels de méthodes sans polluer le code métier.
  • Sécurité : Vérifier les permissions avant l'exécution.
  • Gestion des transactions : Démarrer/valider des transactions automatiquement.
  • Gestion d'erreurs : Intercepter et gérer les exceptions de manière centralisée.
  • Caching : Mettre en cache les résultats de méthodes coûteuses.
  • Synchronisation : Gérer l'accès concurrent aux ressources partagées.

GUIDE MÉTHODOLOGIQUE - DÉCISIONS RAPIDES
1. SOLID – Principes de conception

Méthodologie : Pour chaque classe/interface, pose-toi la question « qui change et pourquoi ? ». Si plus d'une raison de changer → SRP. Si tu modifies du code existant pour ajouter une fonctionnalité → OCP cassé (utilise héritage/interfaces). Si une sous-classe override pour « ne rien faire » → LSP violé. Si une classe implémente une méthode vide d'une interface → ISP violé. Si un module métier importe une classe concrète de bas niveau → DIP violé (injecte une abstraction).

Pièges :

  • ISP ≠ SRP : ISP cible les interfaces (point de vue client), SRP cible les classes (cohésion interne).
  • LSP ne se résume pas à « hériter sans erreur compilateur » : il faut que le comportement reste substituable (Autruche/Oiseau, Carré/Rectangle).
  • DIP ≠ Dependency Injection. DIP = principe (dépendre d'abstractions), DI = technique d'implémentation.
  • OCP ne veut pas dire « jamais toucher au code », mais « ne pas modifier les modules stables et validés ».

Décisions rapides :

  • « Plusieurs interfaces simples > une grosse interface » → ISP
  • « Une seule raison de changer » → SRP
  • « Dépendre des abstractions » → DIP
  • « Sous-type substituable » → LSP
  • « Étendre sans modifier » → OCP
2. Styles Architecturaux

Méthodologie :

  • 1. Identifier la contrainte dominante : centralisation ? scalabilité ? découplage ? temps réel ?
  • 2. Classer en monolithique / modulaire / distribué.
  • 3. Si distribué : choisir entre centralisé (Client-Server, Primary-Secondary, Broker), décentralisé (P2P), réactif (EDA), ou orienté service (SOA).
  • 4. Justifier en citant 2 avantages concrets + 1 désavantage assumé.

Pièges :

  • L'examen 2023 interdit explicitement « modulaire / distribué / multitiered » comme réponse → il faut nommer un style précis (P2P, EDA, etc.).
  • Confondre Broker et SOA : Broker = intermédiaire de routage, SOA = exposition de services.
  • Pipe-Filter ≠ MVC : Pipe-Filter est unidirectionnel et non interactif.
  • Blackboard a un contrôleur (souvent oublié), ce n'est pas juste un tableau partagé.

Décisions rapides :

  • Partage de fichiers sans serveur central, scalable → P2P (ex : BitTorrent)
  • Capteurs IoT temps réel, découplage producteur/consommateur → Event-Driven (EDA)
  • Calcul parallèle big data → Primary-Secondary (MapReduce)
  • Intégration de systèmes hétérogènes via registre de services → Broker ou SOA
  • Système legacy critique, sécurité maximale, équipe restreinte → Mainframe
3. Cadriciels, Plugiciels & Écosystèmes

Méthodologie : Pour distinguer framework vs plugin vs librairie, pose-toi 2 questions :

  • 1. Qui appelle qui ? Si le code te rappelle → framework (IoC). Si tu l'appelles → librairie. Si le système hôte fonctionne sans → plugin.
  • 2. Le système peut-il tourner sans ce composant ? Oui → plugin. Non → framework.

Pièges :

  • Une librairie n'est pas un framework (pas d'IoC).
  • Plugin = SRP (mission unique), pas IoC.
  • Frozen Spots = parties immuables, Hot Spots = points d'extension (pas l'inverse).
  • AngularJS et Django sont des frameworks, pas des écosystèmes.

Décisions rapides :

  • « Le système m'appelle » → Framework
  • « Ajout sans modification du noyau » → Plugin (architecture microkernel)
  • « Mainteneur change l'API → milliers d'apps cassent » → problème de breaking changes dans un écosystème
4. Big Data & NoSQL

Méthodologie :

  • 1. Vérifier les 3V : si au moins un dépasse les capacités SGBDR → Big Data.
  • 2. Identifier la forme des données : clé simple ? document hétérogène ? colonnes massives ? relations complexes ?
  • 3. Appliquer CAP : choisir 2 des 3 (souvent AP en NoSQL).
  • 4. Si cohérence forte requise → ACID (SQL). Si haute dispo + scalabilité → BASE (NoSQL).

Pièges :

  • CAP : P ne se sacrifie pas dans un système distribué — si tu sacrifies P, ce n'est plus distribué.
  • BASE ≠ ACID : BASE accepte la consistance éventuelle.
  • MapReduce : l'input est statique (la phrase « input dynamic » est fausse).
  • Document store ≠ relationnel : pas de JOIN côté serveur.
  • « Reflète un changement immédiatement » → ACID (cohérence forte), pas BASE.

Décisions rapides :

  • Blog avec attributs variables, recherche texte/tags → Document Store (MongoDB)
  • Sessions, paniers, cache → Clé-Valeur (Redis)
  • Logs massifs, séries temporelles → Wide-Column (Cassandra)
  • Réseau social, recommandations → Graphe (Neo4j)
  • Streaming d'événements → Kafka
5. SOA & Mécanismes de Communication

Méthodologie :

  • 1. Latence ? Si minimale + appels directs → RPC/gRPC.
  • 2. Sécurité entreprise + transactions ? → SOAP (WS-Security).
  • 3. Web public, scalable, cache ? → REST.
  • 4. Frontend mobile flexible, sur-fetch à éviter ? → GraphQL.
  • 5. Toujours préciser : stub, interface, SLA, découvrabilité (UDDI).

Pièges :

  • GraphQL ≠ RPC : GraphQL est une query language, RPC = exécution distante.
  • « SOAP = XML standardisé » → vrai (réponse correcte).
  • « REST URI uniques » → vrai, pas partagés.
  • REST n'a pas de contrat formel standard (vs WSDL/SOAP).
  • gRPC = HTTP/2 + Protobuf, pas REST.

Décisions rapides :

  • Télémédecine, appels directs synchrones, faible latence → RPC (ou gRPC)
  • Web public + cache + scaling → REST
  • Banque, contrats stricts, sécurité → SOAP
  • Mobile, sur-fetch problématique → GraphQL
6. EDA & Patrons de Concurrence

Méthodologie :

  • 1. Choisir le patron de synchronisation selon contexte : Singleton → Double-Checked Locking ; appels récursifs → Thread-Safe Interface.
  • 2. Pour le modèle de concurrence : Active Object (asynchrone), Monitor (synchrone), Half-Sync/Half-Async (mixte), Leader/Followers (thread pool).
  • 3. Pour le handling d'événements : Reactor (sync, app fait l'I/O), Proactor (async, OS fait l'I/O), ACT (token de contexte), Acceptor-Connector (séparation connexion/traitement).

Pièges :

  • Reactor vs Proactor : Reactor = l'app fait l'I/O après notification ; Proactor = l'OS fait l'I/O et notifie à la fin.
  • ACT = « cookie » pour retrouver le contexte dans une réponse asynchrone.
  • Acceptor-Connector = essentiel pour P2P orienté connexion.
  • Leader/Followers > Half-Sync/Half-Async en perf (pas de file d'attente entre threads).
  • Scoped Locking seul + récursion = deadlock garanti.

Décisions rapides :

  • App P2P de partage de fichiers → Acceptor-Connector : sépare connexion du traitement
  • Beaucoup de connexions simultanées asynchrones → Proactor
  • Quelques connexions, traitement immédiat → Reactor
  • Réponse différée à corréler → ACT
7. Anti-patrons

Méthodologie :

  • 1. Niveau : système → architectural ; classes → conception ; méthode → code smell.
  • 2. Identifier le principe brisé (SOLID) pour orienter le refactoring.
  • 3. Refactoring patterns : Extract Class/Method, Move Method, Introduce Parameter Object, Replace Conditional with Polymorphism, Hide Delegate.

Pièges :

  • Swiss Army Knife = interface trop polyvalente (réponse correcte).
  • Jumble = fusion horizontal/vertical (couches MVC + services), pas Swiss Army.
  • Cover Your Assets = doc excessive, pas un anti-patron technique.
  • Stovepipe = migration local→distribué ratée.
  • Divergent Change (1 classe, N changements) ≠ Shotgun Surgery (1 changement, N classes).

Décisions rapides :

  • « Interface trop chargée, multi-usages » → Swiss Army Knife
  • « Vue qui contient du SQL direct » → Jumble (DIP cassé)
  • « 12 fichiers à toucher pour 1 changement » → Shotgun Surgery
  • « Réinventer ce qui existe » → Reinvent the Wheel (NIH)
  • « Système local migré sans abstraction » → Stovepipe
8. Blockchain

Méthodologie :

  • 1. Vérifier les 4 propriétés : immuabilité, transparence, décentralisation, consensus.
  • 2. Choisir le consensus : PoW (sécurité, énergie) vs PoS (efficacité) vs PBFT (privé).
  • 3. Calcul du seuil byzantin : n ≥ 3f + 1 pour tolérer f traîtres.
  • 4. Identifier la chaîne : publique (Bitcoin/Ethereum) vs privée (Hyperledger).

Pièges :

  • Trilemme Blockchain = Sécurité, Scalabilité, Décentralisation (réponse correcte).
  • ≠ CAP (C, A, P) — qui est pour les BD distribuées.
  • ≠ ACID (transactions BD).
  • Attaque 51% : peut réordonner / double-dépenser, ne peut pas voler des coins existants.
  • PoS vulnérabilité = Nothing-at-stake, pas 51%.

Décisions rapides :

  • B2B contrôlé, haut TPS → Hyperledger Fabric
  • Smart contracts généraux → Ethereum
  • Monnaie pure → Bitcoin
  • Tolérer 1 traître → minimum 4 nœuds
9. Microservices

Méthodologie :

  • 1. Identifier les Bounded Contexts (DDD) → 1 contexte = 1 service.
  • 2. Appliquer Two-Pizza Team : équipe ≤ 12 personnes par service.
  • 3. Pour migrer : Strangler Application — ne jamais réécrire en big-bang.
  • 4. Choisir la communication : sync (HTTP/REST) pour requête-réponse, async (Kafka/RabbitMQ) pour découplage.

Pièges :

  • Strangling the Monolith = remplacer incrémentalement (réponse correcte). Pas « composer », pas « construire ».
  • Microservices ≠ SOA : Microservices = échelle app, BD par service ; SOA = échelle entreprise, BD partagée.
  • Kafka = streaming (event-based), pas RabbitMQ.
  • RabbitMQ = MQ asynchrone, pas streaming.
  • Nanoservices = anti-patron (More the Merrier).
  • Adopter sans CI/CD ni tests = Trying to fly before walking.

Décisions rapides :

  • Goulot I/O sur S3, latence inter-service → revenir au monolithe (cas Amazon Prime Video)
  • Échec d'un service ne doit pas tout casser → Concevoir pour l'échec + circuit breaker
  • Pile technologique partagée imposée → mauvaise foi (gouvernance décentralisée)
10. AOP – Programmation Orientée Aspects

Méthodologie :

  • 1. Identifier les CCC (cross-cutting concerns) : logging, sécurité, transactions, cache, perf.
  • 2. Définir un Pointcut = expression qui sélectionne des Join Points.
  • 3. Écrire l'Advice = code à exécuter (Before/After/Around).
  • 4. Encapsuler dans un Aspect.
  • 5. Le Weaver tisse au compile-time, load-time ou runtime.

Pièges :

  • « construct that allows injection of CCC » + « pattern expression » → réponse Aspect, Pointcut.
  • Joinpoint ≠ Pointcut : Joinpoint = point d'exécution unique ; Pointcut = expression qui matche plusieurs joinpoints.
  • Advice ≠ Aspect : Advice = code ; Aspect = module qui contient Pointcut + Advice.
  • Un Aspect ne s'instancie pas explicitement (géré par le runtime).
  • Un Aspect concret ne peut pas être hérité.

Décisions rapides :

  • « Logging dispersé partout » → AOP avec Pointcut sur toutes les méthodes publiques (homogène)
  • « Validation différente par méthode » → crosscutting hétérogène
  • « Transaction auto avant/après méthode service » → Advice Around

Q6 — Mythes et réalités sur AOP

Q : Le flot de contrôle du programme est difficile à suivre. R : Effectivement. Mais c'est ce qu'on souhaite ! On veut travailler à un niveau d'abstraction plus élevé. Et le polymorphisme ?

Q : AOP ne règle pas de nouveaux problèmes. R : En effet. On ne cherche pas à résoudre de nouveaux problèmes, mais à résoudre les problèmes actuels de façon plus efficace.

Q : AOP encourage les mauvais designs. R : Faux. AOP nous permet de corriger des problèmes difficilement corrigibles en OO, mais la bonne conception OO des classes issues du domaine du problème reste nécessaire.

Q : AOP peut être utile, mais la définition correcte d'interfaces est tout aussi efficace. R : Faux. Pensez à l'exemple de « logfile » : l'utilisation d'une librairie pour enregistrer les traces d'exécution nécessite quand même une invocation dans chaque fonction dont on veut garder la trace.

Q : AOP ne fait que « patcher » la conception des classes du domaine. R : Faux. AOP permet d'augmenter la modularité, la compréhensibilité et la traçabilité du code.

Q : AOP brise l'encapsulation. R : Vrai, mais d'une façon systématique et contrôlée. On peut briser l'encapsulation d'une classe à partir d'un aspect. AOP est donc très puissant, mais ce pouvoir doit être utilisé correctement.

Q : AOP va remplacer l'OO. R : Faux. Les concepts issus du domaine du problème continuent d'être modélisés avec des objets. AOP est utile pour modéliser les fonctions transversales qui, autrement, nécessiteraient des modifications à plusieurs classes du domaine.

Q : AOP n'est pas mature. R : AspectJ en est à la version 1.9.23 (intégré à Eclipse, NetBeans, JBuilder, Emacs JDDE). AspectC++ 2.2 est disponible (intégré à Eclipse). AOP est disponible dans : Java, C#, C/C++, PHP, Haskell, JavaScript, Matlab, Perl, Prolog, Python, Ruby, XML.

Section 6 — Télémédecine (10 pts)

Question : Select the most suitable SOA communication mechanism (SOAP, REST, RPC, or GraphQL) for the telemedicine scenario (real-time, quick/reliable/direct method invocations, synchronous, secure, low-latency). Justify your choice.

Correction : RPC (Remote Procedure Call).

Justification détaillée : Communication en temps réel : RPC permet des interactions instantanées via des appels de procédure directs. Appels de méthode rapides et directs : RPC abstrait l'appel réseau pour qu'il ressemble à un appel de fonction local, simplifiant les interactions entre les modules de santé. Interactions synchrones fluides : RPC est intrinsèquement synchrone et bloquant par défaut (requête-réponse), assurant la cohérence des échanges entre modules. Fiabilité : RPC garantit la tolérance aux pannes, la livraison sécurisée des messages et les accusés de réception. Sécurité : RPC supporte le chiffrement et l'authentification (TLS, mTLS pour gRPC), critiques pour les données de santé (PHI/HIPAA). Latence minimale : RPC a une conception légère (notamment gRPC avec Protobuf et HTTP/2), bien plus performant que SOAP/XML pour les communications inter-modules. Facilité d'intégration : gRPC est compatible avec un grand nombre de langages et plateformes.

Pourquoi pas les autres ? SOAP : très standardisé et sécurisé, mais le format XML est verbeux → latence et débit moins bons que RPC. REST : architecture orientée ressources via HTTP, idéal pour les APIs publiques basées sur des ressources, mais moins adapté à des invocations de méthodes directes entre modules internes en temps réel ; surcoût HTTP/JSON par rapport à gRPC. GraphQL : excellent pour réduire l'over-fetching dans des applications mobiles ou des UIs complexes, mais ses requêtes sont conçues côté client et son évaluation côté serveur introduit du surcoût ; il n'est pas optimal pour des appels de méthode directs synchrones entre microservices internes à faible latence.

Conclusion : RPC (idéalement gRPC) est le mécanisme SOA le plus adapté à la télémédecine décrite : appels de méthode rapides, synchrones, sécurisés, à très faible latence, avec un excellent support multi-langage pour les différents modules.
