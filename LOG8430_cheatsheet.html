SOLID — 5 Principes
S — SRP Une classe = une seule responsabilité → pas de god class.
O — OCP Ouvert à l'extension, fermé à la modification → abstract/interface pas if/else.
L — LSP Sous-classe remplaçable par superclasse sans casser le comportement.
I — ISP Plusieurs interfaces simples > une grosse → clients implémentent seulement ce qu'ils utilisent.
D — DIP Dépendre des abstractions (interfaces), jamais des implémentations concrètes.

Exam : "Plusieurs interfaces simples" → ISP  |  "Inversion de contrôle" → DIP  |  "Unique responsabilité" → SRP

Framework vs Plugin vs Écosystème
Framework Squelette générique semi-implémenté. Inversion de Contrôle (IoC) : le framework appelle ton code. Développeur implémente les détails dans la structure imposée.

Théorème CAP

Système distribué → au plus 2/3 propriétés simultanément :

C — Cohérence Tout nœud retourne la donnée la plus récente.
A — Disponibilité Toute requête reçoit une réponse (pas forcément à jour).
P — Partition Tolerance Système fonctionne malgré perte de comm réseau.

⚠ P ne peut JAMAIS être sacrifié dans un vrai système distribué (partitions réseau inévitables).

EDA — Architecture Orientée Événements
Événement N'importe quelle action pertinente (clic GUI, changement données).
Message Notification qui suit l'événement → envoyée aux consommateurs.
Bus d'événements Canal/tampon où événements sont stockés et transmis aux listeners.

4 Défis EDA :

1. Accès/config service → Wrapper Façade, Component Configurator, Intercepteur, Extension Interface

2. Synchronisation → Scoped Locking, Strategized Locking, Thread-Safe Interface, Double-Checked Locking

3. Concurrence → Objet Actif, Objet Moniteur, Half-Sync/Half-Async, Leader/Followers

4. Gestion des événements → Reactor, Proactor, ACT, Acceptor-Connector

Patrons d'Accès et Config Service
Wrapper Façade Encapsule fonctions/données d'API non-OO dans interface OO.
Component Configurator Bind/unbind implémentations à runtime sans recompilation.
Intercepteur Services transparents ajoutés au cadre, déclenchés automatiquement sur événement.
Extension Interface Composant exporte plusieurs interfaces → évite surcharge d'interface.
Patrons de Synchronisation
Scoped Locking Verrou acquis/libéré automatiquement à entrée/sortie de portée (RAII-like).
Strategized Locking Paramétrer mécanisme de synchro pour protéger sections critiques.
Thread-Safe Interface Minimise surcharge verrouillage ; évite réacquisition de verrou déjà détenu.
Double-Checked Locking Réduit surcharge synchro pour sections critiques n'acquérant verrou qu'une fois.
Patrons de Concurrence
Objet Actif Découple invocation de méthode de son exécution (appel async).
Objet Moniteur Garantit qu'une seule méthode à la fois s'exécute dans un objet → exclusion mutuelle.
Half-Sync/Half-Async Couche async + couche sync séparées → simplifie prog sans réduire perf.
Leader/Followers Threads se relayent ; un leader surveille sources d'événements → distribue aux followers.

Patrons de Gestion d'Événements — Les 4
Reactor Boucle d'événements qui dispatche I/O synchrones aux handlers appropriés. Détection async → dispatch sync. Gestion réactive.
Proactor Lance opérations asynchrones et traite résultats à la fin. Exécution simultanée sans attente active. → idéal haute concurrence (P2P, transferts fichiers simultanés).
ACT Asynchronous Completion Token. Permet à l'app de démultiplexer et traiter efficacement les réponses des opérations async.
Acceptor-Connector Découple connexion et initialisation des pairs coopérants du traitement effectué une fois connectés. Sépare connection vs processing.
PatronSync/AsyncRôle clé
ReactorDétection async, dispatch syncBoucle dispatch I/O
ProactorTotalement asyncTraite résultats d'opérations lancées
ACTAsyncDémultiplexage réponses
Acceptor-ConnectorIndépendantSépare connexion du traitement

⚠ P2P file-sharing (haute concurrence, I/O simultanés) → Proactor (gestion async complète)

AOP — Programmation Orientée Aspects
Cross-cutting concern Préoccupation transversale coupant plusieurs modules : logging, caching, sécurité, transactions, monitoring.
Aspect Construction qui permet l'injection de cross-cutting concerns sans modifier le code directement.
Advice La méthode qui implémente la tâche transversale (ex: logique de logging concrète).
Pointcut Expression de motif (pattern) qui identifie OÙ le Advice doit être invoqué.
Joinpoint Un point spécifique d'exécution du programme (appel méthode, accès champ…). ≠ Pointcut (le motif)

⚠ Exam : "Aspect, Pointcut" → réponse correcte. Pas Joinpoint (≠ expression de motif/pattern).

Pourquoi AOP : Évite duplication des cross-cutting concerns. Séparation préoccupations sans polluer code métier.

Template Q3 — Plugin vs Framework (6 pts)

Plugin :

  • Ajoute ou étend fonctionnalités spécifiques sans modifier le code central.
  • Introduit nouvelles capacités mais repose sur l'infrastructure de l'application hôte.
  • Flux de contrôle déclenché par événements : l'application hôte décide quand les fonctions du plugin sont exécutées.

Framework :

  • Fournit plan structuré avec fonctionnalités prédéfinies que les développeurs étendent.
  • Suit le principe de l'Inversion de Contrôle (IoC) : le framework gère le cycle de vie et le flux de contrôle de l'application.
  • Les développeurs implémentent les détails spécifiques à l'intérieur de la structure imposée.

Différence clé : Framework = IoC (framework appelle ton code). Plugin = hôte appelle le plugin sur événement.

Template Q4.1 — P2P File Sharing Style (8 pts)

Choix : Peer-to-Peer (P2P)

  • Permet partage direct de fichiers entre utilisateurs sans serveur central → contrainte principale satisfaite.
  • Scalabilité naturelle : chaque nouvel utilisateur apporte ressources supplémentaires au réseau.
  • Réduit coûts liés à l'infrastructure centralisée.
  • Évite les goulots d'étranglement associés aux serveurs centralisés.
  • Tous les pairs sont égaux → pas de point de défaillance unique.
Template Q4.2 — Event Handling P2P (4 pts)

Choix : Proactor

  • Gestion asynchrone complète des opérations I/O (transferts fichiers simultanés).
  • Très performant pour systèmes à forte concurrence : nombreux transferts simultanés entre pairs.
  • Exécution simultanée sans attente active → pas de blocage de thread pendant transfert.
  • Idéal P2P : chaque pair est producteur et consommateur simultané de fichiers.
  • Lance opérations async et traite résultats une fois terminés.
Template Q5 — NoSQL Blog (7 pts)

Choix : Base de données orientée Documents (ex: MongoDB)

  • Données semi-structurées avec attributs variables → schéma flexible (titre, content, auteur, tags, commentaires, images, vidéos).
  • Recherche textuelle efficace via indexation avancée full-text.
  • Recherche par tags → indexation sur champs de tableau dans document.
  • Applications riches en contenu : document = article complet récupéré en une requête.
  • Hétérogénéité des données entre articles → schéma flexible idéal.
Template Q6.1 — Télémédecine SOA (10 pts)

Choix : RPC (Remote Procedure Call)

  • Communication temps réel : appels directs immédiats → interactions instantanées cruciales pour soins patients.
  • Appels méthode directs : simplifie interactions entre modules santé (récupération données, rendez-vous, consultations).
  • Fiabilité : tolérance pannes, livraison sécurisée des messages, accusés de réception.
  • Sécurité : chiffrement + authentification pour données patients sensibles.
  • Faible latence : conception légère vs SOAP (trop verbeux XML) pour expérience optimale.
  • Intégration : compatible multi-langages et plateformes pour modules hétérogènes.

≠ SOAP (verbeux, lourd, pas adapté temps réel) · ≠ REST (resource-based, pas sync direct procédure) · ≠ GraphQL (queries flexibles, pas appels directs procédure)

⚠ Pièges Classiques à l'Examen

1. Multitier ≠ Layered seul : Multitier = MVC déployé sur réseau ✅ distribué. Layered seul = non distribué → refusé.

2. Jamais "distribué" tout court : toujours préciser style exact (P2P, Broker, SOA, EDA…).

3. Kafka ≠ RabbitMQ pour streaming événements : Kafka = event-driven ; RabbitMQ = messaging avec routage.

4. GraphQL ≠ RPC : GraphQL = queries flexibles sur schéma ; RPC = appels de procédures distants synchrones.

5. Partition Tolerance toujours requis en système distribué (jamais sacrifiable).

6. Blockchain trilemme ≠ CAP : Sécurité/Scalabilité/Décentralisation (pas C/A/P).

7. Pointcut ≠ Joinpoint : Pointcut = expression motif WHERE ; Joinpoint = point précis d'exécution.

8. SOA → toujours mentionner le type de communication (SOAP/REST/RPC/GraphQL).

9. MapReduce dataset = STATIQUE, pas dynamique.

10. Proactor pour haute concurrence async (P2P) ; Reactor pour détection sync dispatch.

11. Swiss Army Knife = interface complexe trop de cas (pas Cover Your Assets, pas Jumble).

12. ISP = plusieurs interfaces simples > une grosse (pas DIP, pas SRP).

Microservices — Communication & Patterns
Indépendance données Chaque μservice a copie locale de ses données → pas de dépendance inter-service pour les données.
Async (Event-Driven) Producteur émet événement → consommateurs réagissent indépendamment → haute scalabilité.
Sync (RPC/REST) Appel direct → attente réponse → couplage temporel → pour cas temps réel critique.
Strangling Remplacement progressif fonctionnalité par fonctionnalité. Monolithe coexiste avec μservices pendant migration.
Définition Service Web (SOA)

Un service est : Indépendant (pas de dépendance externe) · Atomique (produit résultats complets) · Public (expose fonctionnalité/données explicitement).

Publie une interface → boîte noire pour les clients. Compose des architectures modulaires et distribuées.

SOA vs Microservices : SOA = portée entreprise (services partagés entre apps). Microservices = portée application (chaque app a ses propres services).
